Settings & Admin — Functional Specification

1. Settings — My Profile (/settings)

The Settings page is available to all authenticated users. It is divided into three sections: Profile, Change Password, and App Configuration.

1.1 Profile section

FieldEditableNotes
NameYesSaved via PUT /api/v1/users/me; displayed in the sidebar and admin user table
EmailNoRead-only; shown for reference. Contact an admin to change email.
RoleNoRead-only badge (Admin / Analyst / Viewer); assigned by admin only

1.2 Change password

Three input fields: Current Password, New Password (minimum 8 characters), and Confirm New Password. Saved via PUT /api/v1/users/me/password. The backend verifies the current password before accepting the change. If the current password is incorrect, a 400 error is returned and displayed below the form.

1.3 App configuration (all users)

Controls the file paths used by the portfolio tracker. These settings are stored in /app/uploads/app_config.json inside the container and take precedence over environment variable defaults.

FieldDefaultDescription
Source File Path/app/investment_data/Investment tracking.xlsxFull container path to the read-only Excel source file. This is mapped from the host via the Docker volume mount.
Working Copy Path/app/uploads/investment_tracking.xlsxFull container path where the writable working copy is stored. This path must be inside a writable volume.
  • Test button — verifies the source path is readable within the container; shows the file size on success, or an error message if the path does not exist or is not readable
  • Changes take effect immediately for all subsequent portfolio reads (no restart required)
Paths entered here are container-internal paths. The host directory is mapped via the Docker volume mount in docker-compose.yml. If you need to change the host-side directory, update the volume mount and restart the container — do not change the source path here unless you have updated the volume mount first.

2. Admin — User Management (/admin/users)

Visible in the sidebar only for users with the admin role. All user management API endpoints are protected by require_admin on the backend — even if someone navigates directly to the URL, all API calls will return 403.

2.1 User list

ColumnDescription
NameUser's display name
EmailLogin email address
RoleRole badge: Admin / Analyst / Viewer
StatusActive (green check) or Disabled (grey)
Last LoginDate and time of last successful login; blank if never logged in
ActionsEdit pencil · Set Password key · Delete trash (with confirmation)

2.2 Create user

The + Create User button at the top opens a modal with fields for Name, Email, Role (dropdown: admin / analyst / viewer), Password, and Confirm Password. The new user is created as active immediately. They can log in as soon as the admin saves the form.

2.3 Edit user

The Edit modal allows the admin to change the user's display name, role, and active status. Deactivating a user prevents login without deleting their data (action plans, weekly scans, etc.).

2.4 Set password

Allows the admin to force-set a user's password without knowing the current password. Useful for account recovery when a user has forgotten their password and the forgot-password email workflow is unavailable.

2.5 Self-protection rules

To prevent admins from accidentally locking themselves out:

  • An admin cannot change their own role (the role field is disabled in the Edit modal for the current user)
  • An admin cannot deactivate their own account (the active toggle is disabled)
  • An admin cannot delete themselves (the Delete button is hidden for the current user row)
  • These restrictions are enforced both in the UI and on the backend (the delete endpoint returns 400 if user_id == current_user.id)

3. Documents (/settings/documents)

An in-app documentation browser accessible to all authenticated users. The documentation content is read from the ./docs directory, which is mounted read-only into the backend container.

3.1 Tree navigation

A left sidebar shows the documentation tree loaded from GET /api/v1/docs-content/manifest. The manifest is a JSON file (docs/manifest.json) describing the tree structure of sections and pages. Groups expand and collapse. Clicking a leaf node loads that document. The first leaf is auto-selected on page load.

3.2 Content view

The selected document is rendered as formatted Markdown using react-markdown with the remark-gfm plugin (for tables, strikethrough, and task lists). Code blocks get syntax highlighting. The rendered content matches the styling of the rest of the application.

3.3 Export options

Four export buttons provide different ways to save the documentation for offline access or sharing:

ButtonOutputDescription
Export MD.md fileDownloads the current page's raw Markdown source
Export HTML.html fileDownloads the current page as a self-contained HTML file with inline styles
Export All (MD)Single .md fileFetches every document in the manifest, concatenates with section headers, downloads as one file
Export All (HTML)Single .html fileSame but wrapped in a styled HTML template with CDN-rendered Markdown
The HTML export files use CDN resources (marked.js and github-markdown-css) so they render correctly when opened in any browser without a web server — useful for sharing documentation with people who don't have access to the application.
