Database Migrations — Technical Guide

This document covers the Alembic migration workflow, how to write and run migrations, the Redis key schema, and safe schema evolution rules. Source files: backend/alembic/, backend/alembic.ini.

1. Alembic Setup

1.1 File Layout

backend/

Alembic uses a synchronous engine even though the application uses asyncpg. The env.py constructs a sync URL by replacing postgresql+asyncpg:// with postgresql+psycopg2:// (or postgresql://) for migration runs only.

2. Common Alembic Commands

All commands run inside the backend container. Replace inv_backend with your container name if different.

2.1 Check Current Migration State

# Show the current revision applied to the database
docker exec inv_backend alembic current

# Show full migration history (oldest → newest)
docker exec inv_backend alembic history --verbose

2.2 Apply Migrations

# Apply all pending migrations (upgrade to latest)
docker exec inv_backend alembic upgrade head

# Apply exactly one pending migration
docker exec inv_backend alembic upgrade +1

# Upgrade to a specific revision
docker exec inv_backend alembic upgrade a6bcb833f755

2.3 Roll Back Migrations

# Roll back one migration
docker exec inv_backend alembic downgrade -1

# Roll back to a specific revision
docker exec inv_backend alembic downgrade a6bcb833f755

# Roll back ALL migrations (empty database — destructive!)
docker exec inv_backend alembic downgrade base
downgrade base drops all managed tables. Only use this on a development or test database.

2.4 Generate a New Migration

# Auto-generate from model changes (inspect diff vs current schema)
docker exec inv_backend alembic revision --autogenerate -m "add_new_column"

# Create a blank migration (for manual data transforms)
docker exec inv_backend alembic revision -m "backfill_sector_data"
--autogenerate compares SQLAlchemy model metadata against the live database. It detects added/dropped columns and tables but does not detect renames — those must be written manually using op.alter_column or op.drop_column + op.add_column.

4. Real Migration Example

Migration a6bcb833f755 modifies the symbol_notes table: makes note nullable and removes two columns.

def upgrade() -> None:
    # Make 'note' column nullable (was NOT NULL)
    op.alter_column('symbol_notes', 'note',
        existing_type=sa.TEXT(),
        nullable=True)

    # Remove columns no longer needed
    op.drop_column('symbol_notes', 'asset_type')
    op.drop_column('symbol_notes', 'created_at')


def downgrade() -> None:
    # Restore the removed columns with their original definitions
    op.add_column('symbol_notes',
        sa.Column('created_at',
            postgresql.TIMESTAMP(timezone=True),
            server_default=sa.text('now()'),
            nullable=False))
    op.add_column('symbol_notes',
        sa.Column('asset_type', sa.VARCHAR(length=10), nullable=False))

    # Revert note back to NOT NULL
    op.alter_column('symbol_notes', 'note',
        existing_type=sa.TEXT(),
        nullable=False)

5. Common Migration Operations

5.1 Add a Nullable Column

def upgrade() -> None:
    op.add_column('positions',
        sa.Column('strategy_tag', sa.VARCHAR(50), nullable=True))

def downgrade() -> None:
    op.drop_column('positions', 'strategy_tag')

5.2 Add a NOT NULL Column with Default

def upgrade() -> None:
    # 1. Add as nullable first
    op.add_column('positions',
        sa.Column('currency', sa.VARCHAR(3), nullable=True))
    # 2. Backfill existing rows
    op.execute("UPDATE positions SET currency = 'THB' WHERE currency IS NULL")
    # 3. Apply NOT NULL constraint
    op.alter_column('positions', 'currency', nullable=False)

def downgrade() -> None:
    op.drop_column('positions', 'currency')
Never add a NOT NULL column with no default in a single step — this will fail on any table that already has rows.

5.3 Create a New Table

def upgrade() -> None:
    op.create_table('watchlist',
        sa.Column('id', postgresql.UUID(as_uuid=True), primary_key=True,
                  server_default=sa.text('gen_random_uuid()')),
        sa.Column('user_id', postgresql.UUID(as_uuid=True),
                  sa.ForeignKey('users.id', ondelete='CASCADE'), nullable=False),
        sa.Column('symbol', sa.VARCHAR(20), nullable=False),
        sa.Column('created_at', sa.TIMESTAMP(timezone=True),
                  server_default=sa.text('now()'), nullable=False),
    )
    op.create_index('ix_watchlist_user_id', 'watchlist', ['user_id'])

def downgrade() -> None:
    op.drop_index('ix_watchlist_user_id', 'watchlist')
    op.drop_table('watchlist')

5.4 Create an Index Concurrently (large tables)

def upgrade() -> None:
    # CONCURRENTLY avoids full table lock on large tables
    op.execute(
        "CREATE INDEX CONCURRENTLY ix_positions_symbol ON positions (symbol)"
    )

def downgrade() -> None:
    op.execute("DROP INDEX CONCURRENTLY ix_positions_symbol")
CONCURRENTLY cannot run inside a transaction. Alembic wraps migrations in transactions by default. To use it, add with op.get_context().autocommit_block(): or set transaction_per_migration = false in alembic.ini.

5.5 Rename a Column

def upgrade() -> None:
    op.alter_column('positions', 'pnl', new_column_name='net_pnl')

def downgrade() -> None:
    op.alter_column('positions', 'net_pnl', new_column_name='pnl')

6. Migration Rules

#RuleRationale
1Always implement downgrade()Enables rollback during failed deploys
2Test downgrade() locally before mergingA broken downgrade is discovered after — not before — a bad deploy
3Never add NOT NULL without a DEFAULT or two-step backfillExisting rows fail the constraint
4Do not modify a migration after it has been applied to productionAlembic uses revision IDs — modifying a file does not re-run it
5One concern per migration fileEasier to bisect failures and roll back selectively
6Use CONCURRENTLY for indexes on tables >100k rowsAvoids locking users out during index creation

7. Redis Key Schema

Redis is used for three purposes: JWT token blacklisting, password reset tokens, and rate limiting. All keys are set with TTL — no manual cleanup is needed.

Key patternTTLValueSet by
blacklist:{jti}Remaining access token lifetime (seconds)"1" (presence is the flag)POST /auth/logout
pwd_reset:{token}3600 s (1 hour)User email address stringPOST /auth/forgot-password
rate_limit:{ip}60 sRequest count string (incremented per request)Rate-limit middleware

7.1 Token Blacklist Flow

# On logout — blacklist the current access token's JTI
await redis.setex(f"blacklist:{jti}", ttl_seconds, "1")

# On every authenticated request — check blacklist
if await redis.exists(f"blacklist:{jti}"):
    raise HTTPException(401, "Token has been revoked")

7.2 Password Reset Flow

# Generate and store reset token
reset_token = secrets.token_urlsafe(32)
await redis.setex(f"pwd_reset:{reset_token}", 3600, user_email)

# On password reset submission — validate and consume
email = await redis.get(f"pwd_reset:{reset_token}")
if not email:
    raise HTTPException(400, "Invalid or expired reset token")
await redis.delete(f"pwd_reset:{reset_token}")   # one-time use

7.3 Recommended Redis Configuration

# redis.conf (or Docker command args)
maxmemory-policy allkeys-lru   # evict LRU keys when memory is full
maxmemory 256mb                # set appropriate to host memory
If Redis runs out of memory and maxmemory-policy is noeviction (the default), all write operations will fail — including token blacklisting. Always set maxmemory-policy allkeys-lru in production.

8. Schema Evolution Checklist

Before merging any migration, verify each item:

  • Migration file has a correct down_revision pointing to the previous revision
  • upgrade() runs successfully on a local copy of the database
  • downgrade() runs successfully and leaves the schema in the prior state
    • No NOT NULL column added without DEFAULT or two-step backfill
  • Large-table index creation uses CONCURRENTLY
  • SQLAlchemy model is updated to match the new schema
  • Pydantic schemas updated if the changed column is exposed in the API
  • alembic current shows the correct head revision after upgrade
