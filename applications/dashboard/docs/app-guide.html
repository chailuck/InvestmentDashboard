Project Overview

Purpose

InvestmentDashboard (internally called InvestPro) is a full-stack, self-hosted investment dashboard for tracking a personal Thai SET (Stock Exchange of Thailand) portfolio. It provides real-time P&L, advanced charting, AI-assisted analysis, structured trade planning, and a weekly scan discipline tool — all running inside Docker containers on a local machine with no cloud dependency.

All data stays local. The system works behind any external tunnel (Cloudflare, ngrok) without rebuilding the frontend.

Key personas

  • Solo investor — reviews the portfolio once a week, tracks open and closed positions read from a local Excel file, and wants to see cumulative P&L at a glance.
  • Weekly scanner — systematically evaluates every symbol on a watchlist each weekend, assigns colour marks, and promotes promising setups to an action plan.
  • Active planner — maintains purchase watchlists and portfolio-level action plans with specific entry targets, take-profits, and stop-losses.

Non-goals

  • Multi-user portfolio isolation (one shared Excel source of truth)
  • Real-time WebSocket price streaming from the exchange
  • Order execution or brokerage integration
  • Mobile native app

# 2. Create the backend secrets file

# 3. Edit backend/.env — required fields:
#    ANTHROPIC_API_KEY=sk-ant-...         (for AI Copilot)

# 4. Edit docker-compose.yml — update the host volume path for the Excel file:
#    volumes:
#      - /path/to/your/investmentPlan:/app/investment_data:ro

# 6. Open the app
#    App:          http://localhost:3000
#    Backend API:  http://localhost:8000

#    Adminer DB:   docker compose --profile dev up -d  →  http://localhost:8080
The admin account is seeded automatically on first startup using ADMIN_EMAIL / ADMIN_PASSWORD from backend/.env. The seed uses ON CONFLICT DO NOTHING, so it is safe to restart containers.

Admin password reset

If you lose the admin password and cannot log in, reset it via Adminer:

  1. Start Adminer: docker compose --profile dev up -d
  2. Open http://localhost:8080
  3. Connect: System = PostgreSQL, Server = postgres, User = postgres, Password = postgres, Database = investment_db
  4. Navigate to the users table, find your admin row, and update the hashed_password column with a bcrypt hash generated for your new password.
Generate a bcrypt hash: python -c "from passlib.context import CryptContext; print(CryptContext(schemes=['bcrypt']).hash('YourNewPassword'))"

The proxy pattern

All browser API calls go through a Next.js server-side catch-all route at /api/proxy/[...path] (frontend/src/app/api/proxy/[...path]/route.ts). The proxy:

  • Reads BACKEND_URL at runtime (not baked into the bundle)
  • Forwards the request including the Authorization: Bearer header to http://backend:8000/...
  • Buffers the request body as ArrayBuffer before forwarding (fixes streaming body issues)
  • Streams the response body back — supports SSE for the AI Copilot
  • Strips hop-by-hop headers (host, connection, transfer-encoding)
Because the browser only ever talks to the same-origin Next.js server, this setup works correctly behind any Cloudflare Tunnel, ngrok, or other reverse proxy without rebuilding the frontend image. Just change BACKEND_URL at container start time.

Authentication flow

Browser  →  POST /api/proxy/api/v1/auth/login  →  FastAPI
         ←  { accessToken, refreshToken, user }  ←

Browser  →  GET /api/proxy/api/v1/...
            Authorization: Bearer <accessToken>  →  FastAPI (verifies JWT)

On 401:
  Axios interceptor  →  POST /api/proxy/api/v1/auth/refresh
  If success  →  retry original request with new token
  If failure  →  clearAuth()  →  redirect to /login

Token storage: Zustand store persisted to sessionStorage (cleared on tab close). The refresh token is short-lived and rotated on each use.

Portfolio data flow

Excel file (read-only volume mount)
  │  pandas read_excel (openpyxl engine)
  ▼
portfolio_excel.py._load_df()
  │  in-memory cache: { df, mtime }
  │  invalidated when .cache_bust file mtime changes (cross-worker)
  │
  ├── get_positions()          →  yfinance live prices for open positions
  ├── get_daily_performance()  →  grouped by day/week/month, cumsum
  ├── get_performance_by_date()
  ├── get_performance_by_stock()
  └── get_period_transactions()

Frontend:
  TanStack Query (staleTime: 60s, refetchInterval: 60s)
    → portfolioTrackerService
    → /api/proxy/api/v1/portfolio-tracker/...
    → FastAPI → above

The backend runs 4 Uvicorn workers, each with its own in-memory cache. When the user clicks Refresh, the backend copies the source Excel to the uploads volume and writes a .cache_bust timestamp file. Every subsequent _cached() call checks that file's mtime — if newer than the cache, all workers clear and reload independently.

PostgreSQL 16. Schema managed by Alembic (alembic upgrade head). SQLAlchemy Base.metadata.create_all is also called at startup as a development convenience, but Alembic is the canonical migration path. Migration files live in backend/alembic/versions/.

Table: users

All authenticated users. Roles control access to admin endpoints.

CREATE TABLE users (
    id               UUID         PRIMARY KEY DEFAULT uuid_generate_v4(),
    email            VARCHAR(255) NOT NULL UNIQUE,
    name             VARCHAR(255) NOT NULL,
    hashed_password  VARCHAR(255) NOT NULL,        -- bcrypt via passlib
    role             user_role    NOT NULL DEFAULT 'viewer',
                                                   -- ENUM: 'admin' | 'analyst' | 'viewer'
    is_active        BOOLEAN      NOT NULL DEFAULT TRUE,
    created_at       TIMESTAMPTZ  DEFAULT NOW(),
    updated_at       TIMESTAMPTZ  DEFAULT NOW(),
    last_login_at    TIMESTAMPTZ
);
CREATE INDEX ON users (email);
ColumnNotes
idUUID PK, auto-generated
emailUnique; used as the login identifier
hashed_passwordbcrypt hash (passlib CryptContext)
rolePostgreSQL ENUM user_role: admin / analyst / viewer
is_activeSoft-disable without deleting the account
last_login_atUpdated on every successful login

TanStack Query

All server data goes through TanStack Query v5. Default configuration:

  • staleTime: 60_000 — data considered fresh for 60 seconds
  • refetchInterval: 60_000 — auto-polls every 60 seconds for live data pages
  • Mutations invalidate the relevant query key on success

API client

An Axios instance in services/api.ts handles all HTTP calls. Request interceptor attaches Authorization: Bearer <token> from Zustand. Response interceptor catches 401 responses and attempts a token refresh before retrying the original request once. On refresh failure, it calls clearAuth() and redirects to /login.

All frontend API calls go to /api/proxy/... (relative URL), never directly to localhost:8000.

Weekly Scan Module

Overview

The Weekly Scan module supports a disciplined, repeatable workflow: once per week, the analyst creates a dated scan from their watchlists, evaluates each symbol's chart setup, assigns a colour verdict, and promotes the best setups to an action plan. The module is personal — each user maintains independent scans, symbol lists, and notes.

Core concepts

Symbol lists

A user maintains one or more named symbol lists (e.g. "SET50", "Crypto DRs", "US Watchlist"). Each list has a market context that drives Yahoo Finance ticker suffix logic. Lists are ordered by sort_order; symbols within a list preserve the user-defined order.

Scans

A scan is a dated snapshot. Name convention: WEEKLY_SCAN_DD_MM_YYYY (target Saturday). When created, all symbols from all active lists are copied into the scan grouped by list name. Items are not automatically updated when the underlying list changes — use Refresh to merge.

Weekly Review

Overview

The Weekly Review module (/action-plan/review-list) provides a structured post-trade journalling workflow. Each ISO week (Monday–Sunday) has exactly one review record per user. The analyst records which positions were entered or exited during the week, rates the quality of each buy and sell decision with an independent feeling score, and captures the Monday-open / Friday-close price movement for every symbol reviewed.

A review is split into two parts: Part 1 — Trades (stocks that had buy or sell activity this week) and Part 2 — Hold (open positions with no activity, used for passive tracking). Active portfolio positions not yet in Part 1 are surfaced as open suggestions to make it easy to add them as HOLD items.

Core concepts

Review record

One weekly_reviews row per user per ISO week, uniquely constrained by (user_id, week_start). Auto-created by GET /current-week the first time it is accessed. The auto-generated name format is Week {N} ({DD Mon}–{DD Mon YYYY}) and is freely editable.

Item types

TypePurposeTypical data
TRADE Position with buy and/or sell activity this week buy_date, buy_price, buy_size, sell_date, sell_price, sell_size (either or both legs may be populated)
HOLD Open position with no activity this week symbol, entry reference from portfolio DB, notes

Feeling rating (1–5)

Each TRADE item carries two independent ratings — one for the buy decision and one for the sell decision. Ratings evaluate the quality of the decision, not the outcome.

ValueLabelMeaning
5Very GoodClear setup, excellent timing, full conviction
4GoodSolid reasoning, mostly confident
3ModerateReasonable but uncertain; acceptable setup
2BadFOMO or reactive; weak rationale
1Very BadImpulsive; against the plan

  • Displays all past weekly reviews in reverse-chronological order
  • Each row shows: week name, date range, trade count, hold count, buy count, sell count, last-updated timestamp
  • Click a row to open the review detail editor
  • Current Week button calls GET /api/v1/review-list/current-week, auto-creating the review for the current Monday if needed
  • Month filter (3 / 6 / 12 / All) for navigating history

AI Copilot

Overview

The AI Copilot (/ai-copilot) is a streaming chat interface backed by Anthropic's Claude claude-sonnet-4-6 model. The assistant has access to portfolio tools and can answer questions about positions, performance, and market data.

How it works

  1. The browser sends a POST /api/v1/ai/copilot/stream with the user message and session ID
  2. The Next.js proxy streams the text/event-stream SSE response back to the browser
  3. FastAPI runs a tool-use loop (up to 6 iterations) calling portfolio tools as needed
  4. SSE events are emitted for each step: text, tool_start, tool_end, error, done

Available tools

Tool nameWhat it does
get_portfolio_positionsReads all current positions from the Excel file via portfolio_excel.get_positions()
get_live_priceFetches the current price for a symbol via yfinance
get_performance_summaryReturns P&L summary metrics from the portfolio Excel
read_knowledge_docReads a Markdown knowledge base file from /app/investment_agent/knowledge/
run_analysis_scriptRuns a Python analysis script from the knowledge base (subprocess)

Configuration

  • Requires ANTHROPIC_API_KEY in backend/.env; AI Copilot is disabled (returns 503) without it
  • Model: claude-sonnet-4-6
  • AI knowledge base mounted at /app/investment_agent (read-only, path set by INVESTMENT_AGENT_PATH host volume)
Session history is stored in the FastAPI worker's memory (_SESSIONS dict). With 4 Uvicorn workers, sessions are sticky per TCP connection. History does not persist across container restarts.

Backend Dockerfile

Multi-stage build: builder stage installs Python packages; runtime stage copies them to a non-root appuser.

FROM python:3.12-slim AS builder
WORKDIR /build
RUN apt-get update && apt-get install -y gcc libpq-dev curl
COPY backend/requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt

FROM python:3.12-slim AS runtime
RUN addgroup --gid 1001 appgroup && \
    adduser --uid 1001 --gid 1001 --disabled-password --no-create-home appuser
WORKDIR /app
RUN apt-get update && apt-get install -y libpq5 curl
COPY --from=builder /root/.local /home/appuser/.local
COPY backend/ .
RUN chown -R appuser:appgroup /app
USER appuser
ENV PATH=/home/appuser/.local/bin:$PATH
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000", "--workers", "4"]

In development, the ./backend:/app:cached volume mount provides hot-reload without rebuilding the image. Uvicorn's --reload flag watches for file changes.

Frontend Dockerfile

Three-stage build: deps installs npm packages; builder runs next build; runtime uses Next.js standalone output.

FROM node:20-alpine AS deps
WORKDIR /app
RUN apk add --no-cache libc6-compat
COPY frontend/package.json ./
RUN npm install --no-audit --legacy-peer-deps

FROM node:20-alpine AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY frontend/ .
RUN npm run build          # requires output: 'standalone' in next.config.js

FROM node:20-alpine AS runtime
RUN addgroup --gid 1001 nodejs && \
    adduser --uid 1001 --ingroup nodejs --disabled-password nextjs
WORKDIR /app
COPY --from=builder /app/public ./public
COPY --from=builder --chown=nextjs:nodejs /app/.next/standalone ./
COPY --from=builder --chown=nextjs:nodejs /app/.next/static ./.next/static
USER nextjs
ENV NODE_ENV=production PORT=3000
CMD ["node", "server.js"]
The frontend is a production build. TypeScript and CSS changes require rebuilding: docker compose build frontend && docker compose up -d frontend. There is no hot-reload for the frontend container.

Cloudflare Tunnel deployment

Because all browser API calls go through the Next.js proxy route (not directly to port 8000), you only need to expose port 3000 via the Cloudflare Tunnel. The BACKEND_URL env var remains http://backend:8000 — the internal Docker hostname — regardless of the public URL. No frontend rebuild is required when the public URL changes. 

# Cloudflare Tunnel example: expose only the frontend
cloudflared tunnel run --url http://localhost:3000 my-tunnel

Design Decisions

Why Excel as the portfolio source of truth

The Excel file (Investment tracking.xlsx) was the investor's existing workflow before this app was built. Keeping it as the source means no data migration, no double-entry, and the Excel remains the authoritative record. The backend mounts it read-only and copies it to a working location on explicit refresh. The portfolio_db_positions table exists as a future migration path if the Excel dependency is eventually removed.

Why the Next.js proxy pattern

Decoupling the frontend URL from the backend location is critical for a self-hosted tool that may be exposed via different tunnels (Cloudflare, ngrok, reverse proxy). If the browser called http://localhost:8000 directly, that URL would be baked into the JavaScript bundle at build time and would break behind any tunnel. The proxy means the frontend image is built once and the BACKEND_URL env var is changed at container start time to point to the correct internal hostname.

Why weekly scans are user-scoped

The weekly scan is a personal investment discipline tool. Each user maintains independent watchlists, evaluation verdicts, and notes. There is no shared scan or collaborative review workflow by design — the colour marks and strategies reflect individual conviction, not committee consensus. This also simplifies the data model: all scan tables cascade-delete on users.id with no cross-user foreign keys.

Why Alembic for database migrations

Early versions of the project used Base.metadata.create_all at startup to auto-create tables. This is fragile for production: it cannot drop columns, rename tables, or add constraints to existing tables. Alembic's alembic revision --autogenerate command inspects the ORM models and the current database state and generates a versioned migration script, making schema evolution traceable and reversible. The create_all call is retained only as a development convenience for fresh containers that have no migration history.

Why react-grid-layout for the dashboard

The dashboard is intended to be personalised to the investor's own priorities. Some users want the allocation chart prominent; others want the holdings table front-and-centre. react-grid-layout provides drag-and-drop and resize for a 12-column grid with breakpoint support. Layout state is persisted in localStorage so the user's arrangement survives page refreshes and container restarts.

Why Zustand + TanStack Query (not Redux)

Zustand is used only for truly global, non-server state: auth tokens, layout preferences, and UI notifications. All server data (positions, plans, scans) is managed by TanStack Query, which provides caching, background refetch, mutation-based invalidation, and stale-while-revalidate semantics out of the box. This separation avoids the complexity of manually managing loading and error states in a global Redux store for every API call.

Why 4 Uvicorn workers and a file-based cache bust

Python's GIL limits true parallelism, but multiple workers allow concurrent request handling for independent requests (e.g. one user refreshing data while another runs a weekly scan update). The trade-off is that in-memory state (caches, session history) is not shared. The .cache_bust file provides a cheap cross-worker cache invalidation mechanism without requiring Redis pub/sub or a distributed lock.
