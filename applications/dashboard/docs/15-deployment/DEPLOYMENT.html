Deployment Guide

This document is the operational reference for deploying, updating, and rolling back InvestmentDashboard. The stack runs as five Docker containers managed by Docker Compose.

1. Service Architecture

ContainerImagePortRestart policyDeploy strategy
inv_postgrespostgres:16-alpine (pull)5432unless-stoppedNo rebuild needed — stateless image, data in named volume
inv_redisredis:7-alpine (pull)6379unless-stoppedNo rebuild needed
inv_backenddashboard-backend (local build)8000unless-stoppedLive volume mount — changes apply on save
inv_frontenddashboard-frontend (local build)3000unless-stoppedBaked image — rebuild required on source change
inv_admineradminer:latest (pull)8080unless-stoppedDev profile only (--profile dev)
The backend uses a live volume mount (./backend:/app:cached), so Python source changes take effect after Uvicorn hot-reloads — typically within 1–2 seconds, with no container restart needed. The frontend is a compiled Next.js standalone image — every source change requires a rebuild.

2. First-Time Setup

# 1. Clone / navigate to project root
cd applications/dashboard

# 2. Create backend environment file
cp backend/.env.example backend/.env
# Edit backend/.env — set ANTHROPIC_API_KEY, ADMIN_EMAIL, ADMIN_PASSWORD, SECRET_KEY

# 3. Build both service images (first run only — takes ~3 min)
docker compose build

# 4. Start all services
docker compose up -d

# 5. Verify all containers are healthy
docker compose ps
The first startup runs database migrations and seeds the admin user. Check docker compose logs backend --tail 30 if the backend fails its health check on first boot.

4. Standard Frontend Rebuild Procedure

Run these two commands from the applications/dashboard/ directory every time frontend source files change:

# Build new image (runs Next.js production build inside Docker)
docker compose build frontend

# Recreate the container from the new image (other containers unaffected)
docker compose up -d --no-deps frontend

The build output includes TypeScript type checking and ESLint. If either fails, the image is not produced and the old container keeps running.

Verify the new image is live

docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}" | grep frontend
# Expected: inv_frontend   Up N seconds (healthy)   0.0.0.0:3000->3000/tcp
The --no-deps flag prevents Docker Compose from also restarting postgres, redis, and backend when only the frontend changed.

5. Standard Backend Rebuild Procedure

# Only needed when requirements.txt changes or Dockerfile.backend is modified.
# Normal Python source changes DO NOT need a rebuild.

docker compose build backend
docker compose up -d --no-deps backend

Apply database migrations

# Always run after pulling changes that include new Alembic versions
docker exec inv_backend alembic upgrade head

# Verify
docker exec inv_backend alembic current

6. Rolling Back a Bad Frontend Deploy

Docker keeps the previous image layer in cache. To roll back:

# Option A — Revert source, rebuild
git revert HEAD
docker compose build frontend
docker compose up -d --no-deps frontend

# Option B — Tag images before deploying (recommended for production)
docker tag dashboard-frontend dashboard-frontend:backup-$(date +%Y%m%d)
# ... then if needed:
docker tag dashboard-frontend:backup-20260602 dashboard-frontend:latest
docker compose up -d --no-deps frontend

7. Viewing Logs

# Tail all services
docker compose logs -f

# Tail a specific service
docker compose logs -f backend
docker compose logs -f frontend

# Last 50 lines from backend
docker compose logs backend --tail 50

# Show container health status
docker inspect inv_backend --format "{{.State.Health.Status}}: {{(index .State.Health.Log 0).Output}}"

8. Stopping and Restarting

# Stop all (data preserved in volumes)
docker compose down

# Stop and remove volumes (DESTRUCTIVE — wipes database)
docker compose down -v

# Restart a single service
docker compose restart backend

# Force-recreate all containers from current images (no rebuild)
docker compose up -d --force-recreate
