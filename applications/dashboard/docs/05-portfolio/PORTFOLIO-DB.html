Portfolio DB Mode — Functional & Technical Spec

Portfolio DB mode is an alternative to the Excel-backed portfolio tracker. Instead of reading position data from a spreadsheet file, positions are stored directly in PostgreSQL and managed via a full CRUD UI. The two modes share the same portfolio page, performance charts, and API response shapes — only the data source differs.

Mode switching is instant and non-destructive. Switching from DB back to Excel does not delete any DB positions. Switching from Excel to DB does not import your Excel data automatically — positions must be entered via the UI or API.

2.2 Header toolbar behaviour

The three action buttons in the page header (Raw Source, Raw Data, Refresh) are only rendered when the active tab is overview. They are hidden when any other tab — including DB Manager — is active. This prevents accidental Excel refresh while operating in the DB manager.

3. Settings → Portfolio DB Page

Route: /settings/portfolio-db

3.1 Mode toggle

A banner at the top of the page shows the current mode (Excel or Database) and a toggle button to switch. The toggle calls PUT /portfolio-db/mode?mode=db or ?mode=excel. The portfolio tracker page immediately uses the new mode on the next data fetch.

3.2 Position list

A filterable table of all positions for the current user. Columns:

3.3 Add position

A modal form with fields: Symbol, Direction (LONG/SHORT), Status (active/closed), Entry Date (defaults to today), Entry Price, Position Size, SL, TP, Exit Date, Exit Price, Remarks. All numeric fields are optional — a position can be created with only a symbol.

3.4 Edit position

Same modal as Add, pre-filled with the current values. Submits a PUT request to replace all fields.

3.5 Sell a position (full or partial)

A "Sell" button opens a dedicated Sell modal with fields: Quantity (shares to sell), Exit Price, Exit Date (defaults to today), Remarks. The Sell modal shows a real-time P&L preview as the exit price is entered (price diff, percentage, and total net THB).

  • Full sell (quantity = position size): the position's status is set to closed and exit_price / exit_date are recorded in-place.
  • Partial sell (quantity < position size): a new child position record is created (status=closed, with exit fields), and the parent's position_size is reduced by the sold quantity. The parent remains active.

3.6 Undo the last partial sell

When a position has child records (i.e., has been partially sold), an "Undo" button appears. It deletes the most recently created child record and restores the parent's position size. Only the last partial sell can be undone.

3.7 Delete position

Hard-deletes the position. If the position has children, the children are also deleted (cascade). A confirmation dialog is shown before deletion.

3.8 Error surfacing

All four mutation operations (create, update, delete, undo-sell) surface failures via toast.error(). The toast message is taken from e.response.data.detail when the server returns a structured error, falling back to a generic message (e.g., "Failed to add position"). Successful operations show a toast.success() confirmation. Silent failures and browser alert() calls are not used.

5. P&L Calculation (DB mode)

# Active position:   current_price fetched from Yahoo Finance (.BK)
# Closed position:   exit_price used as current_price

diff = (current_price - entry_price)      # for LONG
diff = (entry_price  - current_price)      # for SHORT

net_pnl  = round(diff * position_size, 2)
pnl_pct  = round((diff / entry_price) * 100, 2)   # 0 if entry_price is 0

Live prices for active positions are fetched in parallel using asyncio.gather across all unique symbols. Only the .BK suffix is tried (no fallback to bare ticker) to avoid matching US-listed symbols with the same name.

7. Performance Endpoints (DB mode)

These are the same endpoints as the portfolio tracker — the backend routes requests to either Excel helpers or DB helpers based on the user's portfolio_mode. Response shapes are identical in both modes.

EndpointResponse shape
GET /portfolio-tracker/performance?period=monthly[{date, label, dailyPnl, cumulativePnl}] — closed trades only
GET /portfolio-tracker/performance/by-date?period=monthly[{period, label, net, wins, losses, total, winRate, accumulatedPnl}]
GET /portfolio-tracker/performance/transactions?period_key=2026-05&period=monthly[{symbol, direction, entryDate, exitDate, entryPrice, exitPrice, positionSize, netPnl, pnlPct, sl, tp, remarks}]
GET /portfolio-tracker/performance/by-stock[{symbol, net, investment, currentValue, pnlPct, wins, losses, total, winRate}]
In DB mode, performance data is derived from closed positions only (where status = "closed"). Active positions are included in the positions list and in real-time P&L but not in historical performance charts.

8. Mode-Routing Logic

Every portfolio tracker endpoint reads the user's portfolio_mode from the database before deciding which data source to query:

# portfolio_tracker.py — every endpoint calls this helper
async def _mode(user_id: str, db: AsyncSession) -> str:
    from app.api.v1.endpoints.portfolio_db import get_user_portfolio_mode
    return await get_user_portfolio_mode(user_id, db)

# get_user_portfolio_mode reads users.portfolio_mode — defaults to "excel" if null
async def get_user_portfolio_mode(user_id: str, db: AsyncSession) -> str:
    row = await db.execute(select(User.portfolio_mode).where(User.id == uid))
    mode = row.scalar_one_or_none()
    return mode or "excel"

The routing pattern in each endpoint:

if await _mode(user_id, db) == "db":
    return await list_positions_db(user_id, db, from_date, to_date, status)

# else: Excel path
from app.services.portfolio_excel import get_positions
positions = get_positions(from_date=from_date, to_date=to_date, status=status)

File: frontend/src/app/(dashboard)/portfolio/page.tsx (local component, not exported)

function DbManagerPanel() {
  const { data: mode, isLoading, isError } = useQuery({
    queryKey: ['portfolio-db-mode'],
    queryFn: () => portfolioDbService.getMode(),
    staleTime: 5 * 60_000,   // 5 minutes — lazy, fetched on first tab activation

  if (isLoading) return <Loader2 />

9.4 Row ordering logic — orphan-children fix

The position table groups rows as: parent rows first, then each parent's child rows (partial-sell records) interleaved immediately below their parent. When the status filter hides some parents (e.g., "Active" filter hides a closed parent), the child rows whose parent is outside the filter set become "orphan children" and are appended at the end of the table.

A bug existed in the orphan detection predicate. It compared against a single character of the parent ID instead of the full ID, causing orphaned child rows to be silently omitted rather than appended:

// Buggy (old):
const orphanChildren = positions.filter(p => p.parentId && !childMap[p.parentId.split('')[0]])

// Fixed:
const orphanChildren = positions.filter(p => p.parentId && !childMap[p.parentId])

Effect of the fix: partial-sell child rows whose parent is hidden by the current status filter now appear correctly at the bottom of the table, labelled with the "sold" indicator, instead of being dropped from the view.
