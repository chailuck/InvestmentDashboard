Authentication — Functional Specification

1. Overview

InvestPro uses JWT-based authentication with short-lived access tokens and long-lived refresh tokens. The access token is attached to every API request as an HTTP Authorization: Bearer header. All API endpoints require a valid token except for the auth endpoints themselves and the health check probes.

Token refresh is fully automatic and transparent — the user stays logged in for up to 7 days without re-entering credentials. If the refresh token expires or is revoked, the user is silently redirected to the login page.

2. User roles (RBAC)

There are three roles with increasing levels of access. All roles can access the core portfolio, action plan, weekly scan, and AI features. Only the admin role can manage other users.

RoleDescriptionPermissions
adminFull accessAll endpoints including user management; can create, edit, and delete other users; cannot delete themselves
analystPower userAll non-admin endpoints; can update their own profile and password
viewerRead-onlyAll non-admin endpoints; can update their own profile and password; default role for new users
All roles have equal access to all application features (portfolio, scans, action plans, AI). Role-based restrictions only apply to the admin /admin/users page and the underlying user management API endpoints.

3. Login

Page: /login

  1. User enters their email address and password, then clicks Sign In.
  2. Frontend sends POST /api/v1/auth/login with { "email": "...", "password": "..." }.
  3. Backend validates credentials against the hashed password in the database.
  4. On success: the response contains accessToken, refreshToken, and the user profile object. These are stored in the Zustand auth store (persisted to sessionStorage).
  5. Browser navigates to /dashboard.
  6. On failure: the error message "Invalid credentials. Please try again." is displayed below the form.

Token lifetimes:

  • Access token: 30 minutes — configurable via ACCESS_TOKEN_EXPIRE_MINUTES in backend/.env
  • Refresh token: 7 days — configurable via REFRESH_TOKEN_EXPIRE_DAYS in backend/.env

4. Silent token refresh

When any API call returns a 401 Unauthorized response, the Axios interceptor in frontend/src/services/api.ts automatically handles the refresh:

  1. The interceptor catches the 401 response and checks that it hasn't already retried this request (using the _retry flag).
  2. It sends POST /api/v1/auth/refresh with the stored refreshToken in the request body.
  3. On success: the response returns new accessToken and refreshToken. These are stored and the original request is retried with the new access token.
  4. On failure (refresh token expired or revoked): clearAuth() is called to remove all stored tokens, and the browser redirects to /login.

The entire refresh-and-retry cycle is invisible to the user. They see no interruption or error message.

5. Logout

  1. User clicks the Logout button at the bottom of the sidebar.
  2. Frontend calls POST /api/v1/auth/logout with the current Authorization: Bearer header.
  3. Backend extracts the token's JTI (JWT ID) and stores it in Redis with a TTL equal to the token's remaining lifetime. Any subsequent request using this token will be rejected with 401 even before it expires naturally.
  4. Frontend calls clearAuth() which removes all tokens and user data from Zustand and sessionStorage.
  5. Browser redirects to /login.
Tokens also expire naturally when the browser tab is closed because they are stored in sessionStorage, which is cleared by the browser on tab close.

6. Forgot password

Page: /forgot-password

  1. User enters their email address and clicks Send Reset Link.
  2. Frontend calls POST /api/v1/auth/forgot-password with { "email": "..." }.
  3. Backend generates a UUID reset token and stores it in Redis with a 1-hour TTL, keyed to the user's UUID.
  4. The API response includes { "reset_token": "uuid" } directly in the response body (development mode). This token must be copied from the browser's developer tools Network tab and appended to the reset URL.
Email delivery of password reset links is not yet implemented. The reset token is returned in the API response body as a development convenience. In production, the token should be emailed to the user and the reset_token field removed from the response.

7. Password reset

Page: /reset-password?token=<token>

  1. The URL must include the ?token= query parameter obtained from the forgot-password API response.
  2. User enters a new password (minimum 8 characters) in the password and confirm-password fields.
  3. Frontend calls POST /api/v1/auth/reset-password with { "token": "...", "new_password": "..." }.
  4. Backend looks up the token in Redis. If found, it retrieves the associated user UUID, hashes the new password, and updates the database.
  5. On success: Redis deletes the token (preventing reuse) and the browser navigates to /login with a success message.
  6. On failure (token not found or expired): an error message is shown on the page. The user must restart the forgot-password flow.

8. First-run admin seeding

When the backend container starts for the first time, if ADMIN_EMAIL and ADMIN_PASSWORD are set in the environment, a default admin account is automatically created:

INSERT INTO users (id, email, name, hashed_password, role, is_active)
VALUES (uuid_generate_v4(), :email, :name, :bcrypt_hash, 'admin', true)
ON CONFLICT (email) DO NOTHING;

The ON CONFLICT DO NOTHING clause makes this operation idempotent — it is safe to run on every startup (and on all 4 Uvicorn workers simultaneously). If an account with the given email already exists, the statement does nothing and no error is raised.

After first login, change the admin password via Settings → Change Password. The initial password set in .env is retained in that file and should be replaced with a strong password.

9. Protected routes (frontend)

The dashboard layout component (frontend/src/app/(dashboard)/layout.tsx) checks for auth state on every mount. If no valid auth state is found in the Zustand store, it immediately redirects to /login. This provides a first-line client-side guard.

The backend provides a second, authoritative layer of enforcement on every request via the get_current_user FastAPI dependency. Even if someone bypasses the client-side redirect, every API call will be rejected with 401 if the token is missing, invalid, or expired.

Role-based visibility in the sidebar hides the admin-only items from non-admin users:

// Sidebar.tsx — filter admin-only items based on current user role
SETTINGS_SUB.filter(item => !item.adminOnly || user?.role === 'admin')

This is a UI convenience only. The backend enforces role requirements independently via the require_admin dependency on all admin endpoints.
