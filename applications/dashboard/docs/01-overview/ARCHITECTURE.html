2. The Next.js proxy pattern

All browser API calls go through a Next.js server-side catch-all route at /api/proxy/[...path]. The proxy forwards the full request — including the JWT Authorization header — to http://backend:8000 and streams the response back to the browser.

Why this architecture matters:

  • Works behind any external tunnel (ngrok, Cloudflare Tunnel, Tailscale) without rebuilding the frontend — only BACKEND_URL needs to change
  • No CORS issues because the browser only sees the same-origin Next.js server
  • BACKEND_URL is a runtime environment variable, not baked into the bundle at build time
  • SSE streaming for the AI Copilot passes through unchanged because upstream.body is a ReadableStream

File: frontend/src/app/api/proxy/[...path]/route.ts

const BACKEND = (process.env.BACKEND_URL ?? 'http://backend:8000').replace(/\/$/, '')

async function proxy(req: NextRequest, { params }) {
  const url = `${BACKEND}/${params.path.join('/')}${req.nextUrl.search}`
  const hasBody = !['GET', 'HEAD'].includes(req.method)
  const body = hasBody ? await req.arrayBuffer() : undefined

  // Strip hop-by-hop headers that should not be forwarded
  fwdHeaders.delete('content-length')   // let fetch recalculate for the upstream

  const upstream = await fetch(url, {
    method: req.method,
    headers: fwdHeaders,
    body,
    redirect: 'manual',
  })

  return new NextResponse(upstream.body, {
    status: upstream.status,
    headers: resHeaders,
  })
}
Do not use NEXT_PUBLIC_* variables for API URLs. They are baked at build time into the client bundle and cannot be overridden at runtime. Always use BACKEND_URL which is only read server-side by the proxy route.

3. Authentication flow

┌──────────┐   POST /api/proxy/api/v1/auth/login   ┌────────────────┐
│ Browser  │ ──────────────────────────────────────► │  FastAPI       │
│          │ ◄──────────────────────────────────────  │                │
│          │   { accessToken, refreshToken, user }    │  JWT HS256     │
│          │                                          │  signed tokens │
│          │                                          │                │
│          │   GET /api/proxy/api/v1/...              │                │
│          │   Authorization: Bearer {accessToken}    │                │
│          │ ──────────────────────────────────────► │                │
│          │                                          │  verify_token()│
└──────────┘                                          └────────────────┘

On 401 response (access token expired):
  1. Axios interceptor fires: POST /api/proxy/api/v1/auth/refresh
  2. If success: update stored tokens, retry original request transparently
  3. If failure: clearAuth() removes tokens from sessionStorage → redirect to /login

Tokens are stored in Zustand, persisted to sessionStorage. They are automatically cleared when the browser tab is closed, providing a natural session boundary without requiring explicit logout.

  • Access token lifetime: 30 minutes (configurable via ACCESS_TOKEN_EXPIRE_MINUTES)
  • Refresh token lifetime: 7 days (configurable via REFRESH_TOKEN_EXPIRE_DAYS)

Cross-worker cache invalidation

FastAPI runs 4 Uvicorn worker processes. Each worker has its own in-memory _CACHE dictionary. When the Excel file is refreshed (copied from the source volume), a sentinel file /app/uploads/.cache_bust is written with the current timestamp. Every subsequent call to _cached() checks whether the sentinel file's mtime is newer than the cache entry — if so, the cache entry is evicted and the data is reloaded from the Excel file.

This mechanism requires no inter-process communication and works reliably across all 4 workers with a single filesystem write.

The Anthropic SDK call is synchronous. It is wrapped with asyncio.to_thread() to avoid blocking the async event loop while waiting for the API response.

6. Multi-worker architecture

The backend container starts 4 Uvicorn worker processes defined in docker/Dockerfile.backend:

CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000", "--workers", "4"]

Implications of multi-worker deployment:

  • In-memory state like the AI session store (_SESSIONS dict) is not shared between workers. SSE connections are sticky to one worker for the lifetime of the connection, so this works in practice but sessions do not persist across disconnects or restarts.
  • The .cache_bust sentinel file mechanism provides shared cache invalidation without requiring a shared cache layer.
  • The admin account seed uses INSERT ... ON CONFLICT DO NOTHING, making it safe to execute on all 4 workers simultaneously at startup.
  • Rate limiting via SlowAPI uses client IP as the key and is per-worker. Effective rate is N × workers if the load balancer distributes uniformly. For a local setup this is acceptable.
