4. Backend — core logic

4.1 Auto-creation of current week review

GET /review-list/current-week computes Monday of today's date using _monday(date.today()), then attempts to load an existing review for that Monday. If none is found it calls _create_review() and returns the summary. The operation is fully idempotent — concurrent calls for the same week are safe because the unique constraint on (user_id, week_start) prevents duplicate inserts at the database level.

def _monday(d: date) -> date:
    return d - timedelta(days=d.weekday())   # weekday() == 0 for Monday

def _sunday(monday: date) -> date:
    return monday + timedelta(days=6)

def _week_name(monday: date) -> str:
    sunday = _sunday(monday)
    return f"Week {monday.isocalendar().week} ({monday.strftime('%d %b')}–{sunday.strftime('%d %b %Y')})"

@router.get("/current-week")
async def get_or_create_current_week(user_id: UserId, db: DB) -> dict[str, Any]:
    uid = uuid.UUID(user_id)
    monday = _monday(date.today())

    result = await db.execute(
        select(WeeklyReview)
        .where(WeeklyReview.user_id == uid, WeeklyReview.week_start == monday)
        .options(selectinload(WeeklyReview.items))
    )
    review = result.scalar_one_or_none()

    if not review:
        review = await _create_review(user_id, monday, db)
        # Re-fetch with items loaded after creation
        result = await db.execute(
            select(WeeklyReview)
            .where(WeeklyReview.id == review.id)
            .options(selectinload(WeeklyReview.items))
        )
        review = result.scalar_one()

    return _review_summary(review)

4.2 Sync logic

The sync endpoint (POST /{review_id}/sync) reads all user positions from portfolio_positions_db and upserts TRADE items. The full implementation:

for pos in positions:
    in_week_buy  = bool(pos.entry_date and week_start <= pos.entry_date <= week_end)
    in_week_sell = bool(pos.exit_date  and week_start <= pos.exit_date  <= week_end)

    if not in_week_buy and not in_week_sell:
        continue   # position has no activity in this review week

    pos_key = str(pos.id)
    if pos_key in existing_by_pos:
        # Update: only add data that is not yet present (never overwrite annotations)
        item = existing_by_pos[pos_key]
        changed = False
        if item.buy_date is None and pos.entry_date:
            item.buy_date  = pos.entry_date
            item.buy_price = pos.entry_price
            item.buy_size  = pos.position_size
            changed = True
        if in_week_sell and item.sell_date is None:
            item.sell_date  = pos.exit_date
            item.sell_price = pos.exit_price
            item.sell_size  = pos.position_size
            changed = True
        if changed:
            updated += 1
    else:
        # Insert: always include buy leg from entry data for cost-basis context
        item = WeeklyReviewItem(
            review_id=review_id,
            symbol=pos.symbol,
            item_type="TRADE",
            buy_date=pos.entry_date,           # always from position entry
            buy_price=pos.entry_price,
            buy_size=pos.position_size,
            sell_date=pos.exit_date  if in_week_sell else None,
            sell_price=pos.exit_price if in_week_sell else None,
            sell_size=pos.position_size if in_week_sell else None,
            source_position_id=pos.id,
            sort_order=max_order + added,
        )
        db.add(item)
        added += 1

Key design points:

  • Upsert key is source_position_id — only items with a matching source_position_id are considered for update
  • Buy leg is always populated from position entry data (even when entry was before the review week) so that sell-only records still show their cost basis
  • Sell leg is populated only when exit_date is within the review week — not simply because the position is closed
  • Existing field values are never overwritten — only null fields are filled in on update. This preserves any user corrections made after a previous sync
  • User-facing annotations (buy_reason, buy_feeling, sell_reason, sell_feeling) are never touched by sync

4.3 Price refresh

The refresh endpoint (POST /{review_id}/refresh-prices) fetches Monday open and Friday close prices from Yahoo Finance. All symbols are treated as Thai SET equities and the .BK suffix is appended. Each symbol's fetch runs in a thread via asyncio.to_thread to avoid blocking the async event loop on the synchronous yfinance call.

def _week_prices(symbol: str) -> tuple[float | None, float | None]:
    ticker = f"{symbol}.BK"
    try:
        df = yf.Ticker(ticker).history(
            start=fetch_start,            # review.week_start.isoformat()
            end=fetch_end,                # (review.week_end + timedelta(days=1)).isoformat()
            interval="1d",
            auto_adjust=False,
            back_adjust=False,
        )
        if df.empty:
            return None, None
        opens  = df["Open"].dropna()
        closes = df["Close"].dropna()
        open_p  = float(opens.iloc[0])   if len(opens)  >= 1 else None
        close_p = float(closes.iloc[-1]) if len(closes) >= 1 else None
        return open_p, close_p
    except Exception:
        return None, None

# Symbols fetched sequentially via asyncio.to_thread (one thread per symbol)
for sym in symbols:
    prices[sym] = await asyncio.to_thread(_week_prices, sym)

Important details:

  • End date is exclusive — yfinance treats the end parameter as exclusive, so week_end + 1 day is passed to ensure Friday is included in the data window
  • In-progress weeks — closes.iloc[-1] is the latest available close, which will be Wednesday's close (or most recent trading day) when run mid-week
  • Errors are silently swallowed — any yfinance exception returns (None, None); the item's prices are not modified if both values are null
  • Only non-null prices update the item — if yfinance returns an open but no close (or vice versa), only the available value is written

4.4 _item_dict serialiser

All item API responses are produced by _item_dict(). This is where week_change_pct is computed — it is never stored in the database and exists only in the API response:

def _f(v: Any) -> float | None:
    return float(v) if v is not None else None

def _item_dict(item: WeeklyReviewItem) -> dict[str, Any]:
    open_p  = _f(item.week_open_price)
    close_p = _f(item.week_close_price)
    week_change_pct: float | None = None
    if open_p and close_p and open_p != 0:
        week_change_pct = round((close_p - open_p) / open_p * 100, 2)
    return {
        "id": str(item.id),
        "review_id": str(item.review_id),
        "symbol": item.symbol,
        "item_type": item.item_type,
        "buy_date":   item.buy_date.isoformat()  if item.buy_date  else None,
        "buy_price":  _f(item.buy_price),
        "buy_size":   item.buy_size,
        "sell_date":  item.sell_date.isoformat() if item.sell_date else None,
        "sell_price": _f(item.sell_price),
        "sell_size":  item.sell_size,
        "week_open_price":  open_p,
        "week_close_price": close_p,
        "week_change_pct":  week_change_pct,      # computed — not stored
        "buy_reason":  item.buy_reason,
        "buy_feeling": item.buy_feeling,
        "sell_reason": item.sell_reason,
        "sell_feeling": item.sell_feeling,
        "source_position_id": str(item.source_position_id) if item.source_position_id else None,
        "sort_order": item.sort_order,
        "updated_at": item.updated_at.isoformat(),
    }

The _f() helper converts SQLAlchemy Numeric values (which arrive as Python Decimal objects) to float. JSON serialisation of Decimal would raise a TypeError without this conversion.

4.5 _item_sort_key

TRADE items in the detail response are sorted by the date most relevant to the review week, not by database insertion order. The sort key function:

def _item_sort_key(item: WeeklyReviewItem, week_start: date, week_end: date) -> date:
    """Sort priority: sell_date in-week > buy_date in-week > buy_date (any) > sentinel."""
    in_week = lambda d: d is not None and week_start <= d <= week_end
    if in_week(item.sell_date):
        return item.sell_date   # sold this week — sort by exit date
    if in_week(item.buy_date):
        return item.buy_date    # bought this week — sort by entry date
    return item.buy_date or date(9999, 12, 31)   # prior-week entry — sort by buy_date, nulls last

This ordering surfaces the most recent weekly activity first: exits during the week appear before entries during the week, which appear before pre-existing positions. HOLD items are sorted alphabetically by symbol, independently of TRADE items.

4.6 Patch semantics

The PATCH /{review_id}/items/{item_id} endpoint uses Pydantic's model_fields_set to distinguish between a field being explicitly set to null versus a field being absent from the request body. This is critical for feeling ratings, which must be clearable to null:

@router.patch("/{review_id}/items/{item_id}")
async def patch_item(review_id, item_id, body: ReviewItemPatch, user_id, db):
    ...
    # Reasons: treat empty string as null, but only update if field was sent
    if body.buy_reason is not None:
        item.buy_reason = body.buy_reason or None    # "" → null
    if "buy_feeling" in body.model_fields_set:
        item.buy_feeling = body.buy_feeling          # explicit null clears the rating
    if body.sell_reason is not None:
        item.sell_reason = body.sell_reason or None
    if "sell_feeling" in body.model_fields_set:
        item.sell_feeling = body.sell_feeling

Sending { "buy_feeling": null } clears the buy feeling because "buy_feeling" is present in model_fields_set. Sending { "buy_reason": "strong breakout" } without buy_feeling leaves the feeling untouched. Reason fields use a simpler rule: not-in-payload (Pydantic defaults to None) is treated the same as "do not update", but an empty string sent explicitly is normalised to null.

item_type validator

@field_validator("item_type")
@classmethod
def validate_type(cls, v: str) -> str:
    if v not in ("TRADE", "HOLD"):
        raise ValueError("item_type must be TRADE or HOLD")
    return v

Feeling validator

Applied to both buy_feeling and sell_feeling in both ReviewItemIn and ReviewItemPatch:

@field_validator("buy_feeling", "sell_feeling")
@classmethod
def validate_feeling(cls, v: Optional[int]) -> Optional[int]:
    if v is not None and v not in range(1, 6):   # 1, 2, 3, 4, 5
        raise ValueError("feeling must be 1–5")
    return v

A null feeling is always valid (unrated state). Values 0, 6, or any non-integer raise a 422 Unprocessable Entity.

Read-only mode

The detail page derives a readOnly boolean from the review's week_end date:

const today   = new Date()
const weekEnd = new Date(review.week_end)
const readOnly = weekEnd < today && weekEnd.toDateString() !== today.toDateString()

When readOnly is true, Add Item and Sync from Portfolio buttons are hidden, all ReasonInput textareas are disabled, and all FeelingPicker buttons are disabled. Refresh Prices remains available to back-fill historical prices.

7. Frontend — component structure

TradeItemRow

Renders one row in the Part 1 TRADE table. Key implementation details:

function TradeItemRow({ item, readOnly, onPatch, onDelete }) {
  // Sell win/loss — null when either price is missing
  const sellIsWin = item.buy_price != null && item.sell_price != null
    ? item.sell_price >= item.buy_price
    : null

  // Sell price colour class
  const sellPriceClass = sellIsWin === null
    ? 'text-ink-secondary'
    : sellIsWin
      ? 'text-gain'      // green
      : 'text-loss'      // red
}

Column order: Symbol | Price (week_close_price ?? week_open_price) | Buy Date | Buy P/Size | Buy Reason | Buy Feeling | Sell Date | Sell P/Size | Sell Reason | Sell Feeling | Mon/Fri/Wk% | Delete.

Sell reason, sell feeling, and sell P/Size are conditionally rendered — they only appear when item.sell_date is non-null. When sell_date is null, those cells display a dash (—).

HoldItemRow

Renders one row in the Part 2 HOLD table. Simpler than TradeItemRow — no sell leg. The buy leg fields are re-labelled "Entry" in the column headers. Only buy_reason (labelled "Notes") and buy_feeling (labelled "Feel") are annotatable.

FeelingPicker

A compact 2-row, 3-column grid of icon buttons (5 feelings displayed as: row 1 = feelings 5, 4, 3; row 2 = feelings 2, 1 with one empty cell). ReasonInput

A <textarea rows={2}> with local state and a dirty-tracking ref. The save fires on blur only when the value has actually changed:

function ReasonInput({ value, placeholder, readOnly, onSave }) {
  const [local, setLocal] = useState(value ?? '')
  const dirty = useRef(false)

  const handleBlur = () => {
    if (dirty.current) {
      onSave(local.trim() || null)   // empty string → null
      dirty.current = false
    }
  }
}

The dirty ref prevents unnecessary API calls when the user clicks into and out of a field without changing anything.

8. Security considerations

  • Authentication — all 12 endpoints require a valid Bearer JWT. The get_current_user_id dependency extracts and validates the token; unauthenticated requests receive HTTP 401.
  • User-scoping on reads — all queries include WeeklyReview.user_id == uuid.UUID(user_id). A user cannot read or modify another user's reviews.
  • Ownership check returns 404 not 403 — _get_or_404() raises HTTP 404 ("Review not found") for reviews that exist but belong to a different user. This prevents user enumeration by ID guessing — the caller cannot distinguish "does not exist" from "exists but is not yours".
  • Item-level ownership check — the patch and delete item endpoints perform a separate ownership check on the parent review before loading the item, ensuring no item can be accessed by an authenticated user who does not own the review.
  • Safe FK cascade for deleted positions — source_position_id uses ON DELETE SET NULL. If a portfolio position is deleted, review items are preserved and source_position_id is set to null. This prevents orphaned FK references without data loss.
  • Input validation — all request bodies are validated by Pydantic schemas. item_type accepts only TRADE or HOLD; feelings accept only 1–5 or null; both are enforced at the schema level before any database write.
  • No sensitive data in responses — review items contain trading data only; no user credentials, tokens, or PII are included in any endpoint response.

9. Observability

updated_at propagation

SQLAlchemy's onupdate=func.now() on WeeklyReview.updated_at fires when the review row itself changes, but not when only child rows (WeeklyReviewItem) change. To ensure the parent's updated_at stays accurate for the review list page (which shows last-modified time), all item-level mutations explicitly issue a raw UPDATE:

await db.execute(
    text("UPDATE weekly_reviews SET updated_at = NOW() WHERE id = :id"),
    {"id": review_id},
)

This pattern is used in: sync_from_portfolio, refresh_prices, replace_items, patch_item, delete_item, and add_item. It is the same pattern used in the Weekly Scan module for the same reason.

Review summary counters

The _review_summary() function computes four counts inline from the loaded items — no aggregation column is stored:

def _review_summary(review: WeeklyReview) -> dict[str, Any]:
    trade_count = sum(1 for i in review.items if i.item_type == "TRADE")
    hold_count  = sum(1 for i in review.items if i.item_type == "HOLD")
    buy_count   = sum(1 for i in review.items if i.buy_date is not None)
    sell_count  = sum(1 for i in review.items if i.sell_date is not None)
    ...

This requires eager loading of items with every summary query. The selectinload(WeeklyReview.items) option is included in all list and current-week queries for this reason.

10. Model registration

Both ORM classes must be imported in backend/main.py before Base.metadata.create_all runs at startup:

from app.models.weekly_review import WeeklyReview, WeeklyReviewItem  # noqa: F401
Forgetting this import means neither table will be created by create_all(), and Alembic's autogenerate will also fail to detect the models. The module will fail at runtime with a "relation does not exist" error on the first request to any /review-list endpoint.
