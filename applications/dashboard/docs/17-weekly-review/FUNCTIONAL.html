Weekly Review — Functional Specification

1. Overview

The Weekly Review module (/action-plan/review-list) supports a disciplined post-trade journalling workflow structured around ISO calendar weeks (Monday–Sunday). At the end of each week the user records every transaction that occurred, annotates buy and sell decisions with written rationale and a feeling rating, and documents open positions that were held but not traded. Reviewing these records over time builds self-awareness of decision-making patterns and trading psychology.

The module is personal — each user manages their own reviews independently. One user's reviews are never visible to another user. Each review covers exactly one ISO week and the system enforces a unique-per-user-per-week constraint.

Each review has two structural parts:

  • Part 1 — Transactions This Week: all positions where the user bought, sold, or both during the review week (TRADE items)
  • Part 2 — Open Portfolio: all positions that were held open during the week with no buy or sell activity (HOLD items)

2. Core concepts

2.1 Weekly Review record

A Weekly Review is the top-level record that anchors all items to a specific ISO week. Key properties:

  • One review per user per week — enforced by a unique constraint on (user_id, week_start). Attempting to create a second review for the same week returns HTTP 409.
  • Auto-creation on first access — calling GET /review-list/current-week creates this week's review if it does not already exist, then returns it. The operation is idempotent; calling it multiple times is safe.
  • Week boundaries always align to ISO week days — week_start is always the Monday and week_end is always the Sunday of the requested week, regardless of the date supplied by the caller.
  • Auto-generated name — the default name follows the pattern Week N (DD Mon–DD Mon YYYY), for example Week 24 (09 Jun–15 Jun 2026). The user can rename the review at any time via the header edit.
  • Optional notes field — a free-text area on the review header for overall weekly commentary not tied to individual positions.

2.2 Item types

Every item inside a review is classified as one of two types:

TypeWhen to useLegs available
TRADE Any position where a buy transaction, a sell transaction, or both occurred during the review week. This includes: new positions opened this week; existing positions partially or fully closed this week; positions opened and closed within the same week (round-trips). Buy leg + Sell leg (either or both may be populated)
HOLD Positions that were open throughout the week with no transactional activity — the user simply held them. These appear in Part 2 to give a complete picture of the portfolio and allow mood/notes tracking for held positions. Buy leg only (entry details); sell leg is not applicable
A symbol may appear as either TRADE or HOLD in the same review, but not both. Open suggestions (section 5) exclude symbols already present in the TRADE list.

2.3 Buy and sell legs

Each TRADE item has two independent legs. Both legs are optional — they can be filled independently depending on what happened during the week:

  • Buy leg (buy_date, buy_price, buy_size) — populated when the user opened or added to a position during the week. For items synced from Portfolio DB, this always reflects the original entry date and price even when the entry date is before the review week, so the sell side has its cost-basis reference visible.
  • Sell leg (sell_date, sell_price, sell_size) — populated when the user closed or reduced a position during the week. Only synced when exit_date falls within the review week.
  • Both legs set — valid for round-trip trades opened and closed within the same week.
  • Only buy leg set — position was opened this week and is still held.
  • Only sell leg set — position was closed this week but was originally opened in a prior week (sync will still include the buy leg from the portfolio entry data to give cost-basis context).

For HOLD items, only the buy leg is used and it stores the original entry details (date, price, size) from the portfolio position, not a new transaction.

2.4 Feeling rating system

Each buy leg and sell leg carries an independent feeling rating — a 1–5 score that captures the user's subjective quality assessment of the decision itself, not the financial outcome. A trade can feel "Very Good" even if it results in a loss (the process was correct but the market moved against it), and vice versa.

ValueLabelIconWhen to use
5Very GoodChevronsUpThe decision was fully aligned with the plan; analysis was thorough; execution was precise
4GoodChevronUpThe decision was generally sound with minor reservations
3ModerateMinusMixed signals; decision was acceptable but uncertain
2BadChevronDownThe decision was poor in process — chased price, ignored stop, acted on emotion
1Very BadChevronsDownSignificant process failure — violated plan, no analysis, revenge trading
Feeling ratings are independent for buy and sell legs. A user who made a well-planned entry (feeling 4) but panicked on exit (feeling 2) can record both accurately. A null rating means the leg has not been evaluated yet.

Clicking an active feeling button deselects it (sets the value to null), allowing the user to clear a rating they no longer agree with. Changes to feelings save immediately without requiring a blur or submit action.

2.5 Week price snapshot

Every item stores two optional price points that describe how the symbol performed during the review week:

  • week_open_price — the Monday opening price for the review week, fetched from Yahoo Finance via the Refresh Prices action
  • week_close_price — the Friday closing price for the review week. For an in-progress week (where Friday has not yet occurred), the latest available close is used instead

A third derived value, week_change_pct, is computed on-the-fly and never stored in the database:

week_change_pct = (week_close_price - week_open_price) / week_open_price * 100

The UI displays all three values stacked in a "Mon / Fri / Wk%" column. The percentage is green when positive and red when negative. If either price is null the column shows dashes.

All symbols in the review are treated as Thai SET equities for price fetching — the .BK suffix is appended automatically. See section 3.4 for refresh behaviour.

3. User workflows

3.1 Accessing this week's review

The current week's review is accessible from two entry points:

  • Sidebar widget — the dashboard sidebar shows a summary card for the current week's review (trade count, hold count). Clicking it navigates directly to the review detail editor.
  • Review list page — navigating to /action-plan/review-list shows all past reviews and automatically opens this week's review. If no review exists for the current week, one is created on arrival.

Auto-creation is invisible to the user — the detail page loads immediately with the new (empty) review. The auto-generated name can be changed at any time by editing the review header.

3.2 Review list page

The review list page (/action-plan/review-list) shows all of the user's past reviews in reverse chronological order (newest first). Each row in the list displays:

  • Week name (e.g. Week 24 (09 Jun–15 Jun 2026))
  • Week date range
  • Trade count / Hold count
  • Buy count / Sell count
  • Created and last-updated timestamps

A month filter above the list allows the user to restrict the view to reviews from the last 3, 6, or 12 months, or show all reviews (the default). Clicking any review row navigates to its detail editor.

3.3 Sync from Portfolio

The Sync from Portfolio button (available on the detail page when the review is not read-only) automatically populates Part 1 with TRADE items drawn from the Portfolio DB module. The sync logic works as follows:

  1. The backend scans all of the user's positions in portfolio_positions_db.
  2. A position is included if its entry_date falls within the review week or its exit_date falls within the review week (or both).
  3. For each matching position, a TRADE item is created (or updated if one already exists, matched by source_position_id):
    • The buy leg is always populated from the position's entry date and price, even when the entry was in a prior week — this provides cost-basis context next to the sell.
    • The sell leg is populated only when exit_date falls inside the review week.
  4. Items are upserted by source_position_id — running sync multiple times will not create duplicate items. Existing annotation data (reasons, feelings) is never overwritten by a sync.
  5. The response contains { added: N, updated: M } to tell the user how many items were newly created versus refreshed.
After syncing, use Refresh Prices to populate the week price snapshot for all synced symbols in one step.
Sync is only available when the review is not in read-only mode. A review becomes read-only after its week_end date has passed.

3.4 Refresh Prices

The Refresh Prices button fetches Monday open and Friday close prices from Yahoo Finance for every symbol currently in the review (both TRADE and HOLD items). Key behaviour:

  • All symbols are treated as Thai SET equities — the .BK suffix is appended automatically (e.g. BH → BH.BK)
  • The yfinance query window is week_start to week_end + 1 day (the end date is exclusive in the yfinance API)
  • The Monday open price is taken from the first available open in the date range; the Friday close is taken from the last available close
  • For an in-progress week (today is before Friday), the Friday close falls back to the latest available close (e.g. Wednesday's close if run mid-week)
  • If Yahoo Finance returns no data for a symbol, that symbol's prices remain null; other symbols are unaffected
  • Refresh Prices is available even on read-only reviews so that historical prices can be back-filled

The response identifies how many items were updated and lists each symbol with the prices retrieved.

3.5 Editing trade items

All annotation fields on a TRADE item are editable inline in the table. The save strategy differs by field type:

  • Reason fields (buy_reason, sell_reason) — text areas that auto-save on blur. The user types freely; when focus leaves the field the current value is saved. Empty strings are saved as null (no reason).
  • Feeling pickers (buy_feeling, sell_feeling) — compact button grids that save immediately on click. No blur required. Clicking the currently active button deselects it (clears the rating).

Structural fields (symbol, dates, prices, sizes) are set at item-creation time and are not editable inline. To correct a structural field, delete the item and re-add it manually or re-sync from Portfolio.

The sell price is colour-coded to immediately signal trade outcome:

  • Green — sell_price >= buy_price (profitable or break-even)
  • Red — sell_price < buy_price (loss)
  • Gray — either price is null (outcome not determinable)

3.6 Adding items manually

The Add Item button opens a modal form. The user selects the item type and fills in the trade details:

  • Symbol — the ticker symbol (auto-uppercased)
  • Type — TRADE (Buy/Sell) or HOLD (open position)
  • Buy leg — date, price, size (always shown; date defaults to today)
  • Sell leg — date, price, size (shown only when type is TRADE; optional)

Manual addition is appropriate when:

  • The position was tracked in an Excel file (not in Portfolio DB) and therefore not picked up by Sync
  • The user wants to add a HOLD item before using the open-suggestions panel
  • A specific position needs to be logged that does not exist in the portfolio system
For positions tracked in Portfolio DB, prefer Sync from Portfolio over manual addition — the sync captures buy/sell legs from the actual position record and maintains the source_position_id back-link for traceability.

3.7 Deleting items

Each item row has a trash icon delete button. Deleting an item is permanent and immediate — there is no confirmation dialog. The review's updated_at timestamp is bumped on every item deletion. Deleted items cannot be recovered; re-running Sync will re-add any Portfolio DB positions that match the review week.

Week price columns

The rightmost data column in the TRADE table shows three stacked values:

  • Mon — week_open_price (Monday open for the review week), displayed in muted gray
  • Fri — week_close_price (Friday close, or latest available), displayed in secondary text
  • Wk% — week_change_pct computed as (Fri − Mon) / Mon × 100, green if positive, red if negative

The "Price" column (second from left) shows week_close_price falling back to week_open_price if the close is null, giving the most current available price at a glance.

5. Open suggestions

Open suggestions are active positions from Portfolio DB (status = 'active') whose symbol does not already appear in the review's TRADE items. They represent the universe of positions the user might want to add to Part 2 as HOLD items.

Suggestions appear in the Part 2 section header as an "Add open (N)" dropdown button. Hovering over the button reveals a list of suggested symbols with their entry prices. Clicking any suggestion creates a HOLD item pre-populated with the symbol, entry price, size, and source_position_id from the portfolio position.

Suggestions are filtered on the client side — they exclude symbols already tracked in either the TRADE table or the HOLD table so that adding one from the dropdown immediately removes it from the suggestion list.

If there are no open suggestions, the dropdown button is hidden entirely to keep the UI clean. Add HOLD items manually if needed.

6. Integration with other modules

6.1 Portfolio DB

The Weekly Review module has a bidirectional relationship with Portfolio DB:

  • Sync (read) — the Sync from Portfolio action reads portfolio_positions_db to auto-populate TRADE items for the review week. The source_position_id foreign key on each item links back to the originating position row.
  • Open suggestions (read) — active positions are presented as HOLD item suggestions on every detail page load.
  • Safe deletion — if a portfolio position is deleted, the corresponding review item's source_position_id is set to null via ON DELETE SET NULL, preserving the review item and all its annotations.

6.2 Sidebar widget

The dashboard sidebar includes a Weekly Review widget that shows a summary of the current week's review: review name, trade count, and hold count. The widget calls GET /review-list/current-week on page load, which creates the review if it doesn't yet exist. Clicking the widget navigates to the review detail editor.

6.3 Analytics modal

Clicking a symbol in the TRADE or HOLD table opens the Analytics modal with a full OHLCV candlestick chart for that symbol. This is the same chart component used throughout the dashboard (EChartsChart with RSI and Stochastic panels), accessible without leaving the review page.

No review exists for the current week GET /review-list/current-week creates one automatically with the auto-generated name. The detail page shows an empty review with both parts empty.
Part 1 is empty The TRADE section shows a prompt: "No trades yet. Use Sync from Portfolio or Add Item to populate."
Part 2 is empty The HOLD section shows a prompt: "No open positions tracked. Use Add open from your portfolio suggestions."
No open suggestions The "Add open (N)" dropdown button is not rendered. All active positions are either already in the review or there are no active positions in Portfolio DB.
yfinance returns no data for a symbol week_open_price and week_close_price remain null for that symbol. The week price columns display dashes. Other symbols in the review are unaffected.
Refresh Prices run on an in-progress week Friday close falls back to the most recent available close (e.g. Wednesday if run mid-week). Monday open is taken from the first row of data for the week.
Symbol appears in both TRADE and HOLD attempts Open suggestions exclude symbols already in the TRADE list. If a HOLD item is manually added for a symbol already in TRADE, there is no server-side constraint preventing it — the user should avoid this manually.
Sync run on a week with no Portfolio DB activity Returns { added: 0, updated: 0 }. No items are created or modified. The review remains unchanged.
Attempt to create a duplicate week review Manual POST /review-list with the same week returns HTTP 409 with detail "A review already exists for this week". The auto-create path (GET /current-week) handles this gracefully by returning the existing review.
Review week has ended (past week) The detail page enters read-only mode: Add Item, Sync from Portfolio, and delete buttons are hidden. Refresh Prices remains available. Existing annotations cannot be changed.
Portfolio position deleted after syncing The review item remains intact. Its source_position_id becomes null (ON DELETE SET NULL). All annotations (reasons, feelings) are preserved.
Months filter applied on the list page Only reviews with week_start >= CURRENT_DATE - INTERVAL 'N months' are returned. The current week's review is always visible regardless of filter if it falls within the range.
