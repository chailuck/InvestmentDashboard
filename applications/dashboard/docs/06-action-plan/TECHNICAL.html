Action Plan — Technical Design

1. Database schema

Three PostgreSQL tables store action plan data. All use UUID primary keys and belong to the same SQLAlchemy model file.

-- Header record for both plan types
CREATE TABLE action_plans (
    id          UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    name        VARCHAR(200) NOT NULL,
    plan_type   VARCHAR(20)  NOT NULL,    -- 'purchase' | 'portfolio'
    created_by  UUID REFERENCES users(id) ON DELETE SET NULL,
    created_at  TIMESTAMPTZ DEFAULT NOW(),
    updated_at  TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX ON action_plans (created_by, plan_type);

-- Purchase plan line items
CREATE TABLE purchase_plan_items (
    id            UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    plan_id       UUID NOT NULL REFERENCES action_plans(id) ON DELETE CASCADE,
    sort_order    INTEGER NOT NULL DEFAULT 0,
    stock         VARCHAR(20)  NOT NULL DEFAULT '',
    current_price NUMERIC(14,4),   -- yfinance snapshot at save time
    size          INTEGER,
    buy_price     NUMERIC(14,4),
    tp            NUMERIC(14,4),
    sl            NUMERIC(14,4),
    strategy      VARCHAR(200),
    created_at    TIMESTAMPTZ DEFAULT NOW(),
    updated_at    TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX ON purchase_plan_items (plan_id);

-- Portfolio plan line items
CREATE TABLE portfolio_plan_items (
    id            UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    plan_id       UUID NOT NULL REFERENCES action_plans(id) ON DELETE CASCADE,
    sort_order    INTEGER NOT NULL DEFAULT 0,
    symbol        VARCHAR(20)  NOT NULL DEFAULT '',
    current_price NUMERIC(14,4),
    size          INTEGER,
    entry_price   NUMERIC(14,4),
    tp            NUMERIC(14,4),
    sl            NUMERIC(14,4),
    order_size    INTEGER,
    created_at    TIMESTAMPTZ DEFAULT NOW(),
    updated_at    TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX ON portfolio_plan_items (plan_id);

Update strategy (PUT)

The PUT endpoint uses a delete-all + bulk-insert strategy within a single database transaction. This avoids partial-update bugs where a stale row could survive, and keeps the code simple at the cost of slightly higher write load.

@router.put("/{plan_id}")
async def update_plan(plan_id: uuid.UUID, body: PlanUpdate, _: UserId, db: AsyncSession):
    # Verify plan belongs to current user
    plan = await _get_plan_or_404(plan_id, db)

    if body.name:
        plan.name = body.name
        plan.updated_at = datetime.now(UTC)

    if plan.plan_type == "purchase":
        await db.execute(delete(PurchasePlanItem).where(PurchasePlanItem.plan_id == plan_id))
        await db.flush()
        for i, item in enumerate(body.purchase_items or []):
            db.add(PurchasePlanItem(plan_id=plan_id, sort_order=i, **item.model_dump()))

    elif plan.plan_type == "portfolio":
        await db.execute(delete(PortfolioPlanItem).where(PortfolioPlanItem.plan_id == plan_id))
        await db.flush()
        for i, item in enumerate(body.portfolio_items or []):
            db.add(PortfolioPlanItem(plan_id=plan_id, sort_order=i, **item.model_dump()))

    await db.commit()

Refresh All Prices — implementation pattern

The refreshAllPrices function is an async handler attached to the Refresh Prices toolbar button. It fetches live prices for all rows concurrently using Promise.all, with per-item error isolation via .catch.

RefreshResult interface

interface RefreshResult {
  total:         number    // rows that had a non-empty stock symbol
  succeeded:     number    // symbols for which a price was returned
  failed:        number    // symbols that returned null or threw
  failedSymbols: string[]  // list of the failed ticker names
}

rowsRef — ref for async access to current rows

The component keeps a rowsRef = useRef(rows) that is updated in the same effect that sets rows state. This is necessary because refreshAllPrices is wrapped in useCallback and reads the row list at call time — if it closed over the rows state variable directly it would capture a stale snapshot from the render in which the callback was created. The ref always reflects the current value regardless of when the callback fires.

// Keep ref in sync with state
useEffect(() => { rowsRef.current = rows }, [rows])

// Async handler reads from ref, not from rows state
const refreshAllPrices = useCallback(async () => {
  const currentRows = rowsRef.current
  ...
}, [])   // empty dependency array — no stale closure on rows

Promise.all fan-out with per-item error isolation

Each row's price fetch is wrapped in its own .catch before being passed to Promise.all. This means a single failed fetch does not reject the whole batch — the failed item resolves to null and is counted in failedSymbols.

const results = await Promise.all(
  symbolRows.map(row =>
    actionPlanService
      .getStockPrice(row.stock.trim())
      .then(price => ({ key: row._key, stock: row.stock, price }))
      .catch(() => ({ key: row._key, stock: row.stock, price: null }))
  )
)

// Apply successful results to rows state
const updates: Record<string, number | null> = {}
const failedSymbols: string[] = []
for (const r of results) {
  if (r.price !== null) updates[r.key] = r.price
  else failedSymbols.push(r.stock)
}
setRows(prev => prev.map(row =>
  row._key in updates ? { ...row, current_price: updates[row._key] } : row
))

formatSymbolList helper

Notification messages that list failed symbols are truncated at 5 to avoid overflowing the UI. The helper returns a comma-separated string with an optional and N more suffix.

const formatSymbolList = (symbols: string[], max = 5): string => {
  if (symbols.length <= max) return symbols.join(', ')
  return `${symbols.slice(0, max).join(', ')} and ${symbols.length - max} more`
}

Portfolio editor merge logic

On mount, the portfolio editor fetches both the saved plan and live positions in parallel, then merges them. The live position data takes precedence for read-only fields; saved plan values fill the editable TP/SL/order_size fields.

useEffect(() => {
  const load = async () => {
    const [plan, positions] = await Promise.all([
      actionPlanService.get(id),
      portfolioTrackerService.getPositions({ status: 'active' }),
    ])
    // Build a lookup from saved items: symbol → {tp, sl, order_size}
    const savedMap = Object.fromEntries(
      (plan.portfolio_items ?? []).map(item => [item.symbol, item])
    )
    // Merge live positions with saved planning data
    const merged = positions.map(pos => ({
      symbol:        pos.symbol,
      current_price: pos.currentPrice,
      size:          pos.positionSize,
      entry_price:   pos.entryPrice,
      tp:            savedMap[pos.symbol]?.tp ?? null,
      sl:            savedMap[pos.symbol]?.sl ?? null,
      order_size:    savedMap[pos.symbol]?.order_size ?? null,
    }))
    setRows(merged)
  }
  load()
}, [id])

4. Sidebar widget — Refresh button

The sidebar renders two compact widgets that surface the latest action plan without requiring the user to open the full editor pages: PurchasePlanWidget and PortfolioWidget. Both are implemented in frontend/src/components/layouts/Sidebar.tsx.

Each widget has its own handleRefresh function. Unlike the full-editor Refresh Prices button — which only updates in-memory price cells — the sidebar handleRefresh also writes the updated payload back to the server, so the plan seen in the widget stays in sync with what the full editor would show.

Fetch-before-write pattern

Because the sidebar uses React Query with a staleTime of 2 minutes, the in-memory cache can hold data that pre-dates edits the user made in the full plan editor. To prevent those stale values from overwriting the server's authoritative state, handleRefresh always fetches the plan directly from the server before constructing the update payload.

// Step 1 — Fetch authoritative server state, bypassing React Query cache.
//           If this fails, the function exits without any write.
const serverPlan = await actionPlanService.get(planId)
shouldInvalidate = true   // server was reached; cache will be invalidated regardless

// Step 2 — Fetch live market prices for all symbols in parallel.
const priceMap = ...

// Step 3 — Build the update payload: spread server fields, overwrite only current_price.
const updatedItems = serverPlan.purchase_items.map(item => ({
  ...item,                // buy_price, sl, tp, strategy — all from server
  current_price: priceMap.get(item.stock.trim()) ?? item.current_price,
}))

// Step 4 — Write back.
await actionPlanService.update(planId, { purchase_items: updatedItems })

// Step 5 — Invalidate cache so the widget re-renders with fresh data.
if (shouldInvalidate) {
  await queryClient.invalidateQueries({ queryKey: ['sidebar-purchase-plan-detail'] })
  await queryClient.invalidateQueries({ queryKey: ['sidebar-purchase-plans'] })
}

The PortfolioWidget follows the same pattern, using serverPlan.portfolio_items and spreading entry_price, sl, and tp from the server response.

shouldInvalidate flag

The flag is declared as let shouldInvalidate = false before the try block. It is set to true immediately after the server fetch succeeds — before the write is attempted. This ensures that even if the subsequent actionPlanService.update call throws, the cache is still invalidated in the finally-equivalent if (shouldInvalidate) block. The result is that the widget always reflects the server state after a user-initiated refresh, regardless of whether the price write succeeded.

Sidebar widget data flow

User clicks Refresh (sidebar widget)
  → actionPlanService.get(planId)        // direct server call — bypasses React Query cache
      ↓ success → shouldInvalidate = true
  → GET /stock-price per symbol (parallel, errors isolated per symbol)
      ↓
  → Build payload: server items spread + current_price overwritten from live price
      ↓
  → actionPlanService.update(planId, payload)   // PUT /action-plans/{id}
      ↓
  → queryClient.invalidateQueries(...)   // always runs if server was reached
      ↓
  → React Query re-fetches; widget re-renders
If the initial actionPlanService.get call fails (network error, 404, 5xx), the function returns immediately. No write is attempted and shouldInvalidate remains false, so neither the server state nor the cache is touched.

Bug fix — sidebar refresh stale-cache overwrite (2026-06-16)

File: frontend/src/components/layouts/Sidebar.tsx
Widgets affected: PurchasePlanWidget.handleRefresh and PortfolioWidget.handleRefresh

The bug

Before this fix, both handleRefresh functions read the plan's line items directly from the React Query in-memory cache (stale time: 2 minutes). The cached snapshot was then used as the base for the update payload that was written back to the server. This created a silent data-loss path:

  1. User opens the full plan editor and edits buy_price, sl, or tp (purchase plan) / entry_price, sl, or tp (portfolio plan), then saves.
  2. Within the React Query stale window (up to 2 minutes), the user clicks Refresh on the sidebar widget.
  3. The sidebar reads the stale cached values — which pre-date the editor save — and writes them back to the server, silently overwriting the user's edits with the stale data.
The overwrite was silent: no error was shown and the widget appeared to refresh normally. The only observable symptom was that edited plan values reverted to their previous state.

The fix

Both widgets were changed to call actionPlanService.get(planId) as the first step of handleRefresh. This is a direct HTTP call that bypasses the React Query cache and retrieves the current authoritative state from the server. The server-returned items are then used as the base for the spread-and-override payload; only current_price is overwritten from the live market price API.

FieldBefore fix (source)After fix (source)
buy_price / entry_priceReact Query cache (up to 2 min stale)Server response from actionPlanService.get
slReact Query cache (up to 2 min stale)Server response from actionPlanService.get
tpReact Query cache (up to 2 min stale)Server response from actionPlanService.get
current_priceLive market price APILive market price API (unchanged)

The shouldInvalidate flag was also introduced to guarantee cache invalidation whenever the server was reached, even if the subsequent write step fails — preventing the widget from continuing to display data that is now known to be out of date.

handleSymbolClick — input validation

Before opening AnalyticsModal, the clicked symbol is validated against a regex to reject malformed input:

function handleSymbolClick(symbol: string) {
  if (/^[A-Z0-9\-\.]{1,12}$/i.test(symbol)) {
    setModalSymbol(symbol.toUpperCase())
  }
}

The pattern accepts 1–12 characters drawn from the SET/DR ticker alphabet (letters, digits, hyphen, dot) and is case-insensitive. Any symbol that fails the check is silently dropped — no error is shown to the user. The validated symbol is stored in upper case in modalSymbol state.

The regex does not enforce SET-specific rules (e.g., maximum suffix length for DR tickers). It is a basic sanity guard, not a domain validator.

AnalyticsModal — dynamic import

AnalyticsModal is loaded with next/dynamic at the WeeklyPlanDashboard level, with ssr: false and a null loading fallback. This defers the modal's JavaScript bundle until first interaction and avoids hydration errors. The modal is wrapped in AnimatePresence to support enter/exit transitions driven by Framer Motion.

weekDates.ts — utility reference

WeekDay interface

export interface WeekDay {
  date:      Date    // JavaScript Date object for the calendar day
  label:     string  // 'Mon' | 'Tue' | 'Wed' | 'Thu' | 'Fri'
  dateLabel: string  // date-fns format 'd MMM', e.g. '16 Jun' (no leading zero)
  isToday:   boolean // true only when this Date matches the system date
}

getCurrentWeekDays()

export function getCurrentWeekDays(): WeekDay[]

Returns an array of exactly five WeekDay objects representing Monday through Friday of the week that contains the current system date. Internally, it calls date-fns/startOfWeek with { weekStartsOn: 1 } (ISO Monday anchor) and then applies addDays(monday, 0..4).

Weekend behaviour

When called on a Saturday or Sunday, the function returns the Mon–Fri of the same ISO week (i.e., the five days that just passed). No isToday flag will be true because neither Saturday nor Sunday is in the returned array.

dateLabel format

Uses the date-fns format token 'd MMM' — a non-padded day number followed by the three-letter English month abbreviation. Examples: '1 Jun', '15 Jun', '2 Jan'. The function handles month and year boundaries correctly (verified by unit tests covering 29 Jun → 3 Jul and 29 Dec → 2 Jan spans).

Test coverage

The 21 unit tests in weekDates.test.ts cover: array shape (length, field types), label order, Monday anchor invariant, day-spacing invariant, isToday correctness for each weekday, Saturday and Sunday behaviour (5-day return, no isToday), dateLabel format and no-leading-zero rule, month-boundary span, year-boundary span, idempotent successive calls, and mutation-isolation of the returned array.

PlanCellDisplay — purchase cell

Accepts a PurchaseItem and an optional onSymbolClick callback. When onSymbolClick is provided, the symbol is rendered as a <button>; otherwise as a <span>. A compact prop (default false) reduces font sizes for use in constrained layouts.

Position bar calculation

const range  = hasSL && hasTP ? item.tp - item.sl : 0
const buyPct = range > 0 && hasBuy
  ? Math.max(0, Math.min(100, ((item.buy_price - item.sl) / range) * 100))
  : null
const curPct = range > 0 && hasCur
  ? Math.max(0, Math.min(100, ((item.current_price - item.sl) / range) * 100))
  : null
const showBar = hasSL && hasTP && (buyPct !== null || curPct !== null)

Both percentages are clamped to [0, 100] so markers never overflow the bar even when the price is outside the SL–TP range.

Accepts a PortfolioItem and an optional onSymbolClick callback. The P&L percentage is derived client-side:
