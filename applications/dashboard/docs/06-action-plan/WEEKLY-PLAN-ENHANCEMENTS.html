Weekly Plan Dashboard — Enhancements

Version: 1.0    Date: 2026-06-17    Status: Implemented

This document describes three enhancements shipped together to the Weekly Plan Dashboard (the fifth section of /action-plan). Baseline architecture is described in 06-action-plan/TECHNICAL.html § 5; this document records only the delta.

1. Feature overview

#EnhancementPurpose
E-1 Per-day closing prices in week cells Each Mon–Fri cell shows the historical closing price for that date instead of always showing the same live current_price from the plan record. Cells for future dates are blank; today's cell uses the live price from the plan.
E-2 Week selector (previous / next week navigation) Users can scroll backward up to 52 weeks and forward to the current week. Column headers and all price cells update to match the selected week.
E-3 Symbol removed from cell body; moved to row header Each row's symbol is shown once in the sticky row header <th> as a clickable button, not repeated inside every cell. This reduces visual noise and makes the symbol a single reliable click target.

2. Backend — new endpoint

Endpoint definition

AttributeValue
MethodGET
Path/api/v1/action-plans/price-history
AuthenticationBearer JWT (same get_current_user_id dependency as all other action-plan endpoints)
Router filebackend/app/api/v1/endpoints/action_plan.py

Query parameters

ParameterTypeRequiredValidationNotes
symbolsstringYesComma-separated; max 20 items; each item must match [A-Z0-9.\-]{1,20}SET ticker symbols. Duplicates are silently deduplicated.
date_fromISO dateYesFormat YYYY-MM-DD; must be a MondayFirst day of the requested week window.
date_toISO dateYesFormat YYYY-MM-DD; must be exactly 4 calendar days after date_fromLast day of the requested week window (Friday).
The endpoint enforces that date_to - date_from == 4 days exactly. This prevents callers from requesting arbitrarily large date ranges via this endpoint. Requests spanning more than one week must be split by the caller.

Response — HTTP 200

{
  "date_from": "2026-06-16",
  "date_to":   "2026-06-20",
  "prices": {
    "BH":    { "2026-06-16": 122.50, "2026-06-17": 124.00 },
    "KBANK": { "2026-06-16": 141.00 },
    "PTT":   {}
  }
}

The prices object contains one key per requested symbol. Each symbol maps to an object whose keys are ISO date strings and whose values are closing prices as floats. A symbol with no available data for any date in the range returns an empty object ({}), not an error.

TypeScript response type

export type DailyPrices = Record<string, number>        // { "2026-06-16": 122.50 }
export type WeekPriceMap = Record<string, DailyPrices>  // { "BH": DailyPrices }

export interface PriceHistoryResponse {
  date_from: string
  date_to:   string
  prices:    WeekPriceMap
}

Defined in frontend/src/services/actionPlan.ts.

Error responses

HTTP statusConditionBody
400Missing or invalid query parameters (symbol format, date format, date range not exactly 5 days)FastAPI validation error JSON
401Missing or invalid Bearer token{"detail": "Not authenticated"}

Partial failure behaviour

If the data source returns no closing prices for a symbol (e.g., the symbol is delisted, or market data is unavailable for the requested dates), the backend includes the symbol key in prices with an empty object. The frontend treats an absent key and an empty object identically — the affected cells are rendered blank. No error is surfaced to the user for individual symbol failures.

Frontend service method

// frontend/src/services/actionPlan.ts
actionPlanService.getPriceHistory(
  symbols:   string[],
  dateFrom:  string,   // "2026-06-16"
  dateTo:    string    // "2026-06-20"
): Promise<PriceHistoryResponse>

Constructs the symbols query parameter by joining the array with commas. Called once per week view render when the component mounts or when the user changes the selected week.

weekDates.ts — isoDate field added to WeekDay

The WeekDay interface gains one field. This is the only change to weekDates.ts.

export interface WeekDay {
  date:      Date     // JavaScript Date object
  label:     string   // 'Mon' | 'Tue' | 'Wed' | 'Thu' | 'Fri'
  dateLabel: string   // 'd MMM' format, e.g. '16 Jun'
  isToday:   boolean  // true when this date matches system date
  isoDate:   string   // NEW — "YYYY-MM-DD", e.g. "2026-06-16"; used as price map key
}

The existing getCurrentWeekDays() is renamed to getWeekDays(weekOffset: number): WeekDay[]. A weekOffset of 0 returns the current ISO week (Mon–Fri); -1 returns the previous week; positive offsets advance into the future. isToday continues to reflect the system date regardless of offset — it will be false for all cells when a past or future week is selected.

The rename of getCurrentWeekDays to getWeekDays is a breaking change to any other caller of the old function. As of this document no other file imports getCurrentWeekDays directly — confirm before future refactors.

WeeklyPlanDashboard.tsx — week selector state

const [weekOffset, setWeekOffset] = useState<number>(0)
const weekDays = useMemo(() => getWeekDays(weekOffset), [weekOffset])

// Derived date range for price history fetch
const dateFrom = weekDays[0].isoDate   // Monday isoDate
const dateTo   = weekDays[4].isoDate   // Friday isoDate

// Price history query (TanStack Query)
const { data: priceHistory } = useQuery({
  queryKey: ['weekly-price-history', dateFrom, dateTo, symbolList],
  queryFn:  () => actionPlanService.getPriceHistory(symbols, dateFrom, dateTo),
  enabled:  symbols.length > 0,
  staleTime: 5 * 60 * 1000,   // 5 minutes — prices for past weeks are immutable
})

// Build the Map used by WeeklyPlanTable
const priceMap = useMemo(() => {
  const map = new Map<string, Map<string, number>>()
  if (priceHistory) {
    for (const [symbol, dailyPrices] of Object.entries(priceHistory.prices)) {
      map.set(symbol, new Map(Object.entries(dailyPrices)))
    }
  }
  return map
}, [priceHistory])

dayPrice resolution (E-1)

For each cell, the table resolves a dayPrice before rendering:

const dayPrice: number | null = (() => {
  const symbolPrices = priceMap.get(symbol)
  if (!symbolPrices) return null

  const day = weekDays[colIndex]

  // Future date — no price available
  if (!isCurrentWeek && !day.isToday /* always false for non-current week */ ) {
    // Use isoDate comparison against today to determine future
  }
  if (day.isoDate > todayIso) return null

  // Today on the current week — use live price from plan item
  if (day.isToday && isCurrentWeek) return item.current_price ?? null

  // Historical date — use price map
  return symbolPrices.get(day.isoDate) ?? null
})()

The resolved dayPrice is passed to the cell component as a required prop. Cells render a price when dayPrice is a number, or render blank when it is null.

5. Architecture decisions

ADR-WP1: Batch price fetch over per-symbol fetch

AttributeDetail
StatusAccepted
Date2026-06-17
DecidersWeekly Plan Dashboard team

Context

The Weekly Plan Dashboard shows price cells for N symbols × 5 days. Two fetch strategies were evaluated for loading historical prices.

Decision

We will use a single batch endpoint (GET /price-history?symbols=A,B,C&date_from=...&date_to=...) that returns all symbols and all dates in one response.

Alternatives considered

AlternativeProsCons
Per-symbol fetch (N requests, one per row) Simpler endpoint; partial results load progressively N concurrent HTTP requests; N separate TanStack Query cache entries to invalidate; no single coordinated loading state
Batch endpoint (chosen) Single network round-trip; one cache entry keyed on the full symbol list + date range; single loading/error state in the UI All-or-nothing: if the batch request fails the entire price grid is unavailable (mitigated by per-symbol empty-object partial failure within a successful response)

Consequences

  • The symbol list passed as a query parameter must be deterministically ordered to avoid cache key fragmentation from different symbol orderings. The frontend should sort symbols before joining.
  • The 20-symbol cap on the endpoint limits the maximum number of tracked symbols in a single weekly view. This is acceptable for a personal portfolio tool.
  • Cache staleTime for past weeks is set to 5 minutes (longer than the 2-minute stale time for plan data) because historical closing prices are immutable once a trading day closes.

Risks

  • If the symbol list grows beyond 20, the endpoint must be called in batches of 20 and results merged client-side. No such batching is implemented in this version.

ADR-WP2: Price data prop-drilled from Dashboard through Table to Cell

AttributeDetail
StatusAccepted
Date2026-06-17
DecidersWeekly Plan Dashboard team

Context

Price data needed to reach the individual cell components (PlanCellDisplay, PortfolioCellDisplay). Two delivery patterns were considered.

Decision

We will prop-drill the priceMap (and the resolved dayPrice) from WeeklyPlanDashboard through WeeklyPlanTable to each cell. Cell components are purely presentational and receive a resolved dayPrice: number | null scalar — they never initiate network requests.

Alternatives considered

AlternativeProsCons
Cell-level fetch (each cell calls useQuery for its own price) Cells are self-contained; no prop drilling N × 5 separate query subscriptions; each cell has its own loading state, making a unified "loading" skeleton impossible; defeats the batch fetch in ADR-WP1
React Context (price data in a context provider) Avoids prop drilling through Table layer Adds a context boundary for a small two-level tree; overkill for the current component depth; makes data flow less explicit
Prop drilling (chosen) Explicit data flow; cells remain stateless presentational components; single TanStack Query subscription in Dashboard; one coordinated loading state WeeklyPlanTable receives and forwards priceMap without using it directly (pass-through prop)

Consequences

  • WeeklyPlanTable now carries a priceMap prop purely to pass it down. This is acceptable given the shallow depth (Dashboard → Table → Cell).
  • Adding a third table type (if a new plan type is introduced) will require threading the same priceMap prop again.

Risks

  • If component depth increases significantly, the prop-drilling approach should be revisited in favour of Context or a shared query hook.

6. Technical debt register

ItemLocationDescriptionSuggested resolution
Symbol list ordering for cache keys WeeklyPlanDashboard.tsx The symbols array passed to getPriceHistory and used as part of the TanStack Query key must be consistently sorted to prevent cache fragmentation. If symbols arrive in different order across renders, duplicate network requests will be made for the same data. Sort symbols alphabetically before building the query key and the API call parameter: const symbolList = [...symbols].sort().join(',')
No batching when symbol count exceeds 20 frontend/src/services/actionPlan.ts The getPriceHistory endpoint enforces a maximum of 20 symbols per request. The frontend does not implement chunking. If a user tracks more than 20 symbols, the request will fail with a 400 error and all price cells will be blank. In getPriceHistory, split the symbols array into chunks of 20, issue parallel requests, and merge the WeekPriceMap responses before returning.
Position bar uses live price, not dayPrice PlanCellDisplay.tsx, PortfolioCellDisplay.tsx The SL–TP position bar in each cell uses item.current_price (the live price stored in the plan record) rather than dayPrice (the historical price for the selected day). On past weeks, the bar position reflects today's price, not the price on that historical day. Pass dayPrice to the bar calculation in place of item.current_price when dayPrice is non-null and the selected week is not the current week.
getCurrentWeekDays rename breaks callers frontend/src/lib/weekDates.ts The function was renamed from getCurrentWeekDays() to getWeekDays(weekOffset). Any future file importing the old name will fail silently at runtime if TypeScript strict mode is not enforced at the import site. Add a deprecated re-export alias in weekDates.ts and remove it after confirming no other callers exist via a codebase search.
