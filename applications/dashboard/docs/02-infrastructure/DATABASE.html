Database Schema

PostgreSQL 16. The schema is managed by Alembic migrations (alembic upgrade head). SQLAlchemy's Base.metadata.create_all is also called at startup as a development convenience for tables that are not yet covered by a migration, but Alembic is the canonical migration path. Migration files live in backend/alembic/versions/.

1. Extensions

Installed by scripts/init_db.sql which is mounted into the PostgreSQL container's docker-entrypoint-initdb.d/ directory and runs only once on first initialisation:

CREATE EXTENSION IF NOT EXISTS "uuid-ossp";  -- uuid_generate_v4() for primary keys
CREATE EXTENSION IF NOT EXISTS "pg_trgm";    -- trigram indexes for future full-text search

2. Tables

users

Stores all authenticated user accounts. The PostgreSQL ENUM type user_role enforces valid role values at the database level.

CREATE TYPE user_role AS ENUM ('admin', 'analyst', 'viewer');

CREATE TABLE users (
    id               UUID         PRIMARY KEY DEFAULT uuid_generate_v4(),
    email            VARCHAR(255) NOT NULL UNIQUE,
    name             VARCHAR(255) NOT NULL,
    hashed_password  VARCHAR(255) NOT NULL,
    role             user_role    NOT NULL DEFAULT 'viewer',
    is_active        BOOLEAN      NOT NULL DEFAULT TRUE,
    created_at       TIMESTAMPTZ  DEFAULT NOW(),
    updated_at       TIMESTAMPTZ  DEFAULT NOW(),
    last_login_at    TIMESTAMPTZ
);

CREATE INDEX ON users (email);
ColumnNotes
idUUID primary key, auto-generated by uuid_generate_v4()
emailUnique; used as the login identifier; not changeable by the user (admin must update)
hashed_passwordbcrypt hash via passlib; uses CryptContext(schemes=["bcrypt"])
rolePostgreSQL ENUM: admin | analyst | viewer
is_activeSoft-disable without deleting; inactive users cannot log in
last_login_atUpdated on every successful login; displayed in the admin user table

action_plans

Header record for both plan types. The plan_type column distinguishes Purchase plans from Portfolio plans. Line items are stored in two separate child tables. Three additional text columns (notes, set_analysis, ai_recommend) hold analyst and AI-generated content.

CREATE TABLE action_plans (
    id           UUID         PRIMARY KEY DEFAULT uuid_generate_v4(),
    name         VARCHAR(200) NOT NULL,
    plan_type    VARCHAR(20)  NOT NULL,     -- 'purchase' | 'portfolio'
    created_by   UUID         REFERENCES users(id) ON DELETE SET NULL,
    notes        TEXT,                      -- freeform notes for the plan
    set_analysis TEXT,                      -- market analysis text (AI or manual)
    ai_recommend TEXT,                      -- AI recommendation text
    created_at   TIMESTAMPTZ  DEFAULT NOW(),
    updated_at   TIMESTAMPTZ  DEFAULT NOW()
);

CREATE INDEX ON action_plans (created_by, plan_type);
ON DELETE SET NULL means plans are retained even if the creating user is deleted. The plan becomes "unowned" but the data is not lost.
ColumnNotes
plan_type'purchase' | 'portfolio' — determines which child table holds line items
notesFreeform analyst notes for this plan; editable in the plan header panel
set_analysisSET market analysis text; may be AI-generated or manually entered by the analyst
ai_recommendAI recommendation text; populated by the AI assist feature

purchase_plan_items

One row per stock in a Purchase Action Plan. Stores the full trade setup for an intended new entry: target buy price, position size, take-profit, stop-loss, strategy, user rationale, and a trigger flag. The current_price is a snapshot fetched from yfinance at the time the plan was saved.

CREATE TABLE purchase_plan_items (
    id            UUID         PRIMARY KEY DEFAULT uuid_generate_v4(),
    plan_id       UUID         NOT NULL REFERENCES action_plans(id) ON DELETE CASCADE,
    sort_order    INTEGER      NOT NULL DEFAULT 0,
    stock         VARCHAR(20)  NOT NULL DEFAULT '',
    current_price NUMERIC(14,4),    -- snapshot of live price at save time
    size          INTEGER,          -- number of shares
    buy_price     NUMERIC(14,4),    -- intended entry price
    tp            NUMERIC(14,4),    -- take-profit price
    sl            NUMERIC(14,4),    -- stop-loss price
    strategy      VARCHAR(200),     -- free text or preset strategy name
    reason        TEXT,             -- user rationale for this trade idea
    triggered     BOOLEAN NOT NULL DEFAULT FALSE,  -- true once price hits buy zone
    created_at    TIMESTAMPTZ  DEFAULT NOW(),
    updated_at    TIMESTAMPTZ  DEFAULT NOW()
);

CREATE INDEX ON purchase_plan_items (plan_id);
RR (risk/reward ratio) is not stored — it is calculated on the frontend as (TP − BUY) / (BUY − SL) and is display-only.
ColumnNotes
current_priceSnapshot fetched from yfinance at save time; not a live value
reasonUser's rationale for including this stock in the plan
triggeredSet to TRUE once the live price enters the buy zone; used for alert UI highlighting

portfolio_plan_items

One row per open position in a Portfolio Action Plan. Unlike purchase items, the position data (symbol, current price, size, entry price) is fetched live from the portfolio tracker on load and is not the source of truth in the database. Only the user's planning fields (TP, SL, order size) are persisted.

CREATE TABLE portfolio_plan_items (
    id            UUID         PRIMARY KEY DEFAULT uuid_generate_v4(),
    plan_id       UUID         NOT NULL REFERENCES action_plans(id) ON DELETE CASCADE,
    sort_order    INTEGER      NOT NULL DEFAULT 0,
    symbol        VARCHAR(20)  NOT NULL DEFAULT '',
    current_price NUMERIC(14,4),    -- snapshot at last save time
    size          INTEGER,          -- position size from portfolio tracker
    entry_price   NUMERIC(14,4),    -- original entry from portfolio tracker
    tp            NUMERIC(14,4),    -- user-defined target price
    sl            NUMERIC(14,4),    -- user-defined stop-loss
    order_size    INTEGER,          -- shares to buy/sell in the planned action
    created_at    TIMESTAMPTZ  DEFAULT NOW(),
    updated_at    TIMESTAMPTZ  DEFAULT NOW()
);

CREATE INDEX ON portfolio_plan_items (plan_id);

user_symbol_lists

weekly_scan_items

One row per symbol within a scan. All evaluation fields are nullable until the analyst fills them in during the review workflow. A unique constraint prevents duplicate symbols within the same scan.

    buy_price   NUMERIC(14, 4),
    size        INTEGER,

    sl          NUMERIC(14, 4),

CREATE INDEX ON weekly_scan_items (scan_id);

weekly_reviews

Header record for one ISO weekly review per user. One row per user per week; the unique constraint on (user_id, week_start) enforces this at the database level. The review is auto-created on first access to the current-week endpoint so the analyst always has a record to work with.

CREATE TABLE weekly_reviews (
    id          UUID         PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id     UUID         NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    week_start  DATE         NOT NULL,
    week_end    DATE         NOT NULL,
    name        VARCHAR(100) NOT NULL,
    notes       TEXT,
    created_at  TIMESTAMPTZ  DEFAULT NOW(),
    updated_at  TIMESTAMPTZ  DEFAULT NOW(),
    CONSTRAINT uq_review_user_week UNIQUE (user_id, week_start)
);

CREATE INDEX ix_weekly_reviews_user_id ON weekly_reviews (user_id);
ColumnNotes
week_startMonday of the ISO week (always Monday); used as the canonical week identifier
week_endSunday of the same ISO week
nameAuto-generated as "Week {N} ({DD Mon}–{DD Mon YYYY})" on creation; editable by the user
notesAnalyst's free-text summary for the week; appears in the review header

weekly_review_items

One row per position within a weekly review. Supports two item types: TRADE (positions with buy/sell activity this week) and HOLD (open positions with no activity). Both legs (buy and sell) are independently nullable — a trade item may be entry-only (still open at week end) or exit-only (position entered in a prior week).

This table went through three migrations between 2026-06-13 and 2026-06-14. The schema below reflects the current final state after migration d4f8c2e73b1a. See section 6 for the full migration history.
CREATE TABLE weekly_review_items (
    id                  UUID         PRIMARY KEY DEFAULT uuid_generate_v4(),
    review_id           UUID         NOT NULL REFERENCES weekly_reviews(id) ON DELETE CASCADE,
    symbol              VARCHAR(30)  NOT NULL,
    item_type           VARCHAR(10)  NOT NULL,    -- TRADE | HOLD

    -- Buy leg (entry this week)
    buy_date            DATE,
    buy_price           NUMERIC(14,4),
    buy_size            INTEGER,

    -- Sell leg (exit this week)
    sell_date           DATE,
    sell_price          NUMERIC(14,4),
    sell_size           INTEGER,

    -- User annotations
    buy_reason          TEXT,
    buy_feeling         SMALLINT,    -- 1=Very Bad ... 5=Very Good; null = unrated
    sell_reason         TEXT,
    sell_feeling        SMALLINT,    -- 1=Very Bad ... 5=Very Good; null = unrated

    -- Week price snapshot (fetched from yfinance on demand)
    week_open_price     NUMERIC(14,4),   -- Monday open of review week
    week_close_price    NUMERIC(14,4),   -- Friday close of review week

    -- Optional back-link to portfolio DB source position
    source_position_id  UUID  REFERENCES portfolio_positions_db(id) ON DELETE SET NULL,

    sort_order          INTEGER NOT NULL DEFAULT 0,
    created_at          TIMESTAMPTZ  DEFAULT NOW(),
    updated_at          TIMESTAMPTZ  DEFAULT NOW()
);

CREATE INDEX ix_weekly_review_items_review_id ON weekly_review_items (review_id);
ColumnNotes
item_typeTRADE = had buy and/or sell activity during the week; HOLD = open position with no activity this week
buy_date, buy_price, buy_sizeBuy leg fields; null if the position was not entered this week
sell_date, sell_price, sell_sizeSell leg fields; null if the position was not exited this week
buy_feeling / sell_feelingIndependent 1–5 ratings (1 = Very Bad, 5 = Very Good); null = unrated. Originally a single feeling column — split in migration d4f8c2e73b1a (2026-06-14)
week_open_priceMonday open price for the review week; fetched from yfinance via /refresh-prices
week_close_priceFriday close price; falls back to the latest available close for in-progress weeks
week_change_pctComputed, not stored. Derived as (week_close_price − week_open_price) / week_open_price × 100 by the API response serialiser. Appears in API responses but is not a database column.
source_position_idLinks to portfolio_positions_db for auto-synced items; null for manually entered items

4. Redis — non-relational store

Redis is used only for ephemeral data with short TTLs. It is not used for caching query results — that cache lives in the FastAPI workers' in-memory _CACHE dictionary. All Redis keys are prefixed and have TTLs set by the application at write time.

Key patternTTLValuePurpose
blacklist:<token_jti>Remaining token lifetime (seconds)"1"Blacklisted JWT tokens after logout; checked on every authenticated request
pwd_reset:<token>3600 s (1 hour)user UUID stringPassword reset tokens; deleted on successful use

The Redis client is a custom CacheClient wrapper at backend/app/database/redis.py that wraps redis.asyncio.Redis.

5. Model registration

All SQLAlchemy ORM models must be explicitly imported in backend/main.py before Base.metadata.create_all is called during the lifespan startup. Importing a model registers it with the SQLAlchemy mapper even if it is not otherwise used in that file. Missing this import results in the table not being created.

from app.models.user import User                                                          # noqa: F401
from app.models.action_plan import ActionPlan, PurchasePlanItem, PortfolioPlanItem        # noqa: F401
from app.models.weekly_scan import WeeklyScan, WeeklyScanItem, UserScanConfig, UserSymbolList  # noqa: F401
from app.models.weekly_review import WeeklyReview, WeeklyReviewItem                      # noqa: F401
from app.models.portfolio_db import PortfolioDbPosition                                  # noqa: F401
from app.models.symbol_note import SymbolNote                                             # noqa: F401
# dr_mappings model is also registered

async with engine.begin() as conn:
    await conn.run_sync(Base.metadata.create_all)
The # noqa: F401 comment suppresses the "imported but unused" linter warning. These imports are intentional side effects and should not be removed.
