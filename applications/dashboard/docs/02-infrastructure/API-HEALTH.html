API Spec — Health

Objective: Provide liveness and readiness probes for Docker health checks, container orchestrators (Kubernetes), and monitoring systems. These endpoints require no authentication and are excluded from rate limiting and metrics collection.

Endpoints at a Glance

MethodPathAuthPurpose
GET/api/v1/health/liveNoneLiveness — is the process running?
GET/api/v1/health/readyNoneReadiness — can the service handle traffic?

GET /api/v1/health/live

Purpose

Returns immediately if the Uvicorn process is running. Used by Docker HEALTHCHECK and Kubernetes liveness probes. A failure here triggers a container restart.

Sample

GET /api/v1/health/live
# No headers required

HTTP/1.1 200 OK
{ "status": "ok" }

GET /api/v1/health/ready

Purpose

Verifies that both PostgreSQL and Redis are reachable. A failure here means the service should not receive traffic (Kubernetes readiness probe removes the pod from load-balancer rotation).

Sample — all healthy

GET /api/v1/health/ready

HTTP/1.1 200 OK
{
  "status": "ok",
  "db":    "ok",
  "redis": "ok"
}

Sample — database unreachable

HTTP/1.1 503 Service Unavailable
{
  "status": "degraded",
  "db":    "error: connection refused",
  "redis": "ok"
}

Error Handling

503 Service Unavailable — returned by /ready when either the database or Redis connection check fails. The response body always includes individual db and redis keys so the monitoring system can pinpoint which dependency is down.
These endpoints are intentionally excluded from the SlowAPIMiddleware rate limiter and from the Prometheus metrics instrumentator to avoid noise in dashboards. They are also excluded from SecurityHeadersMiddleware CSP headers because they return plain JSON with no browser consumer.
