Docker & Deployment

1. docker-compose.yml — full service definitions

All five services (PostgreSQL, Redis, Backend, Frontend, Adminer) are defined in a single docker-compose.yml at the root of the applications/dashboard directory. They all share the dashboard-net bridge network and communicate by hostname.

version: "3.9"
networks:
  dashboard-net:
    driver: bridge

volumes:
  postgres_data:    # PostgreSQL data files
  redis_data:       # Redis AOF persistence
  uploads_data:     # Working Excel copy + app_config.json + .cache_bust

services:
  postgres:
    image: postgres:16-alpine
    container_name: inv_postgres
    restart: unless-stopped
    networks: [dashboard-net]
    environment:
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres
      POSTGRES_DB: investment_db
    volumes:
      - postgres_data:/var/lib/postgresql/data
      - ./scripts/init_db.sql:/docker-entrypoint-initdb.d/init.sql:ro
    ports: ["5432:5432"]
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres -d investment_db"]
      interval: 10s
      timeout: 5s
      retries: 5

  redis:
    image: redis:7-alpine
    container_name: inv_redis
    restart: unless-stopped
    networks: [dashboard-net]
    command: redis-server --appendonly yes --maxmemory 256mb --maxmemory-policy allkeys-lru
    volumes: [redis_data:/data]
    ports: ["6379:6379"]
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 5s
      retries: 3

  backend:
    build:
      context: .
      dockerfile: docker/Dockerfile.backend
    container_name: inv_backend
    restart: unless-stopped
    networks: [dashboard-net]
    depends_on:
      postgres: { condition: service_healthy }
      redis:    { condition: service_healthy }
    environment:
      APP_ENV: development
      DATABASE_URL: postgresql+asyncpg://postgres:postgres@postgres:5432/investment_db
      REDIS_URL: redis://redis:6379/0
      CORS_ORIGINS: '["*"]'
      LOG_LEVEL: INFO
    env_file: [backend/.env]   # overlays secrets on top of environment block
    volumes:
      - ./backend:/app:cached                              # hot-reload in dev
      - uploads_data:/app/uploads                         # working Excel copy
      - /path/to/investmentPlan:/app/investment_data:ro   # source Excel (ro)
      - /path/to/investmentAgent:/app/investment_agent:ro # AI knowledge base (ro)
      - ./docs:/app/docs:ro                               # documentation
    ports: ["8000:8000"]
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/api/v1/health/live"]
      interval: 15s
      timeout: 10s
      retries: 3
      start_period: 30s

  frontend:
    build:
      context: .
      dockerfile: docker/Dockerfile.frontend
    container_name: inv_frontend
    restart: unless-stopped
    networks: [dashboard-net]
    depends_on:
      backend: { condition: service_healthy }
    environment:
      BACKEND_URL: http://backend:8000   # runtime only — never baked at build
      NODE_ENV: production
    ports: ["3000:3000"]

  adminer:
    image: adminer:latest
    container_name: inv_adminer
    restart: unless-stopped
    networks: [dashboard-net]
    depends_on: [postgres]
    ports: ["8080:8080"]
    profiles: [dev]   # only starts with: docker compose --profile dev up
Update the two host paths (/path/to/investmentPlan and /path/to/investmentAgent) before running docker compose up. If the source Excel path is wrong the portfolio tracker will fail with a 503 error.

2. Backend Dockerfile

A multi-stage build that separates package installation from the runtime image. The builder stage installs all Python packages into /root/.local using the --user flag. The runtime stage copies them into a non-root user's home directory. This avoids installing build tools (gcc, libpq-dev) in the final image.

FROM python:3.12-slim AS builder
WORKDIR /build
RUN apt-get update && apt-get install -y gcc libpq-dev curl
COPY backend/requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt

FROM python:3.12-slim AS runtime
# Create non-root user for security
RUN addgroup --gid 1001 appgroup && \
    adduser --uid 1001 --gid 1001 --disabled-password --no-create-home appuser
WORKDIR /app
RUN apt-get update && apt-get install -y libpq5 curl
# Copy installed packages from builder
COPY --from=builder /root/.local /home/appuser/.local
COPY backend/ .
RUN chown -R appuser:appgroup /app
USER appuser
ENV PATH=/home/appuser/.local/bin:$PATH
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000", "--workers", "4"]

Why multi-stage? The builder stage needs gcc and libpq-dev to compile some Python extensions. Including these in the final image would increase the image size and attack surface. The runtime stage only needs the compiled .so files and the libpq5 runtime library.

3. Frontend Dockerfile

A three-stage build: deps installs npm packages, builder runs next build, and runtime uses Next.js's standalone output mode to produce a minimal server that does not need the node_modules directory.

FROM node:20-alpine AS deps
WORKDIR /app
RUN apk add --no-cache libc6-compat
COPY frontend/package.json ./
RUN npm install --no-audit --legacy-peer-deps

FROM node:20-alpine AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY frontend/ .
RUN npm run build    # produces .next/standalone + .next/static

FROM node:20-alpine AS runtime
RUN addgroup --gid 1001 nodejs && \
    adduser --uid 1001 --ingroup nodejs --disabled-password nextjs
WORKDIR /app
COPY --from=builder /app/public ./public
COPY --from=builder --chown=nextjs:nodejs /app/.next/standalone ./
COPY --from=builder --chown=nextjs:nodejs /app/.next/static ./.next/static
USER nextjs
ENV NODE_ENV=production PORT=3000
CMD ["node", "server.js"]
next.config.js must have output: 'standalone' for this Dockerfile to work. Without it the .next/standalone directory will not be generated and the runtime stage will fail.

5. Common commands

# Build all images and start all services (detached)
docker compose up -d --build

# Rebuild and restart a single service without touching others
docker compose build backend && docker compose up -d backend

# Tail logs from a specific container
docker logs inv_backend -f
docker logs inv_frontend -f

# Access the PostgreSQL CLI inside the container
docker exec -it inv_postgres psql -U postgres -d investment_db

# Access the Redis CLI inside the container
docker exec -it inv_redis redis-cli

# Start with Adminer DB browser on port 8080
docker compose --profile dev up -d

# Stop all services (preserves named volumes and data)
docker compose down

# Stop all services and destroy all data volumes
docker compose down -v
docker compose down -v permanently deletes the PostgreSQL and Redis data volumes. All user accounts, action plans, and weekly scans stored in the database will be lost. Only run this to fully reset the environment.

7. Health checks

Docker Compose uses health checks to sequence service startup correctly. The backend will not start until both Postgres and Redis are healthy. The frontend will not start until the backend is healthy.

ServiceHealth testIntervalStart periodNotes
PostgreSQLpg_isready -U postgres -d investment_db10 s—Checks the database accepts connections
Redisredis-cli ping10 s—Returns PONG when Redis is ready
Backendcurl -f http://localhost:8000/api/v1/health/live15 s30 sReturns {"status":"ok"} once Uvicorn is accepting connections
Frontendnone——Depends on backend service_healthy condition
The 30-second start_period on the backend health check gives Uvicorn time to initialise all 4 workers and run the database startup hooks (schema creation, admin seed) before Docker marks it unhealthy.
