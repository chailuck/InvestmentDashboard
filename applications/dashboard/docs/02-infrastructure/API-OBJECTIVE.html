API Reference — Objective Tab

The Objective API provides read and annotation access to portfolio_positions_db rows for the Objective tab on the Portfolio page. It is the data source for the decision-journaling feature: users can view positions filtered by time window and record a free-text reason and a sentiment feel (1–5) against each trade.

Base path: /api/v1/objective
Authentication: Bearer JWT required on all endpoints.
Implementation: backend/app/api/v1/endpoints/objective.py

1. GET /api/v1/objective

GET List portfolio positions for the Objective tab.

Query parameters

ParameterTypeDefaultDescription
portfolio_idUUID stringnullScope to a specific portfolio. If omitted, returns positions across all portfolios for the authenticated user.
monthsinteger (1–24)nullRolling window: only positions where entry_date ≥ today − N months. Omit for ALL positions. Ignored when no_reason_only=true.
no_reason_onlybooleanfalseWhen true, returns only positions where reason IS NULL OR reason = ''. Overrides months.

Sort order

Results are always sorted by entry_date DESC NULLS LAST, then by exit_date DESC NULLS FIRST — open positions (null exit_date) appear before closed positions that share the same entry date.

2. PATCH /api/v1/objective/{position_id}

PATCH Partial update of reason and/or feel on a single position. Called by the auto-save handler when the user edits the Objective tab inline.

Path parameter

ParameterTypeDescription
position_idUUID stringID of the position to update

Request body

// Update only reason
{ "reason": "Breakout above weekly resistance" }

// Update only feel
{ "feel": 4 }

// Update both
{ "reason": "Strong earnings beat", "feel": 5 }

// Clear reason (explicit null)
{ "reason": null }

// Clear feel
{ "feel": null }
Only the fields present in the request body are written. Omitting a field leaves it unchanged. Sending null explicitly clears the field to NULL in the database.

Field validation

FieldConstraintError
reasonMax 5,000 characters or null422 if exceeds limit
feelInteger in {1, 2, 3, 4, 5} or null422 if outside range (including 0, 6, etc.)

Response — 200 OK

Returns the full ObjectivePositionResponse object for the updated position (same shape as the items in the GET response).

Error responses

StatusCause
401Missing or invalid Bearer token
404Position not found, or position belongs to a different user
422feel outside 1–5, reason exceeds 5,000 chars, or non-UUID path parameter
500Database error

3. Architecture notes

Data ownership

Both endpoints enforce user ownership. GET /objective always applies WHERE user_id = <jwt_sub>. PATCH /objective/{id} uses a compound WHERE id = $1 AND user_id = $2 — returning 404 if either condition fails, preventing IDOR enumeration.

New DB columns (migration e5f6a7b8c9d0)

Migration 20260626_1000_e5f6a7b8c9d0 adds reason TEXT and feel SMALLINT to portfolio_positions_db. Both are nullable; existing rows receive NULL. No index is added (single-user, full-scan cost acceptable).

Excel sync preservation

The sync_excel_positions_to_db function captures existing (reason, feel) values keyed on (symbol, entry_date, position_size) before the delete-and-reinsert cycle, then restores them on the new rows. Positions with identical natural keys may lose one set of annotations (known limitation documented in code).

Frontend integration

The frontend service is at frontend/src/services/objective.ts. The ObjectiveTab component manages a single TanStack Query cache key ['objective-positions', portfolioId, filter]. Individual row patches update the cache entry in-place via queryClient.setQueryData without triggering a full refetch.
